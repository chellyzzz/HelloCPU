## 一、概述

为 hcpu 设计并实现了一套分支预测系统，包含 **BTB (Branch Target Buffer)** 和 **RAS (Return Address Stack)**，结合 **JAL 静态预测**，显著减少了控制流指令的流水线惩罚。

### 性能收益

| 测试 | 原始周期 | 预测器周期 | 提升 |
|------|---------|-----------|------|
| quick-sort | 15,458 | 8,346 | **-46%** |
| recursion | 47,638 | 13,379 | **-72%** |
| bubble-sort | 13,878 | 10,515 | **-24%** |
| select-sort | 10,565 | 8,696 | **-18%** |
| hello | 394 | 334 | **-15%** |

## 二、架构设计

```
IFU 阶段 (PC 选择)
┌─────────────────────────────────────────────────────┐
│  pc_next ─┬──→ ICache ──→ ins[31:0]                │
│           ├──→ BTB lookup ──→ predict_valid,target  │
│           ├──→ RAS lookup  ──→ ras_valid,ras_target │
│           └──→ JAL detect  ──→ jal_target (部分译码) │
│                                                     │
│  next_pc = BTB_hit  ? btb_target  :                 │
│            JAL_det ? jal_target  :                 │
│            RAS_hit ? ras_target  :                 │
│            pc+4                                     │
└─────────────────────────────────────────────────────┘

EXU 阶段 (验证/恢复)
┌──────────────────────────────────────────────────────┐
│  实际分支结果 vs predict_taken → mispredict?          │
│  是 → flush IFU/IDU, 重定向 PC, 更新 BTB/RAS         │
│  否 → 更新 BTB 计数器                                │
└──────────────────────────────────────────────────────┘
```

## 三、BTB — 分支目标缓冲

### 结构

- **条目数**: 64
- **映射方式**: 直接映射 (Direct-mapped)
- **索引**: `pc[7:2]` (6-bit, 每条 4 字节指令一个槽)
- **Tag**: `pc[31:8]` (24-bit)

### 条目格式

```
{ valid[1], tag[24], target[30], counter[2] }
  有效位    PC[31:8]  地址[31:2]   2-bit 饱和计数器
                                 (00=SN,01=WN,10=WT,11=ST)
```

预测跳转条件: `counter[1] == 1`

### 行为

| 场景 | BTB 操作 |
|------|---------|
| **Lookup** | `pc[7:2]` 索引 → tag 比对 → `counter[1]` 决定预测方向 |
| **分支跳转 (update)** | 分配/更新条目, counter → ST, 更新 target |
| **分支不跳 (update)** | counter → SN, 不预测跳转 |
| **首次遇到分支** | BTB miss → 预测不跳, 若实际跳转则下次分配条目 |

### 源文件

`vsrc/ifu/btb.v` — 纯时钟同步模块, 组合逻辑 lookup + 时钟边沿 update

## 四、RAS — 返回地址栈

### 结构

- **深度**: 8 条目, LIFO 栈
- **条目宽度**: 30-bit (地址[31:2])

### 操作规则

| 操作 | 检测条件 (EXU) | 时机 |
|------|------------|------|
| **Push** | `JAL` 且 `rd == x1(ra)` | EXU 提交阶段 |
| **Pop** | `JALR` 且 `rd == x0` 且 `rs1 == x1(ra)` | EXU 提交阶段 |
| **Predict** | `JALR` 且 `rd == x0` (IFU 阶段) | IFU 取指阶段 |

预测时仅读取栈顶 (不弹出) → 弹出在 EXU 阶段执行。

### 注意事项

- RAS 更新在 EXU 阶段 (非推测执行), 避免误预测时状态损坏
- 若栈空时发生 Return → RAS miss → 依靠 EXU mispredict 重定向
- 若嵌套深度超过 8 → 最旧条目被丢弃 (不崩溃)

### 源文件

`vsrc/ifu/ras.v` — 纯时钟同步模块, 组合逻辑读栈顶 + 时钟边沿 push/pop

## 五、JAL 静态预测

### 原理

JAL 指令总是跳转。在 IFU 阶段对 `icache_ins` 做最少量的部分译码:

```verilog
wire is_jal  = (icache_ins[6:0] == 7'b1101111);
wire [31:0] jal_imm = {{12{icache_ins[31]}}, icache_ins[19:12],
                        icache_ins[20], icache_ins[30:21], 1'b0};
wire [31:0] jal_target = pc_next + jal_imm;
```

J-type immediate 提取与 `vsrc/idu/idu.v:106` 完全一致, 经验证 0 次 target mismatch。

### 收益

- JAL 指令 0 周期惩罚 (vs 原始 2-3 周期)
- 无需 BTB 条目存储
- 面积开销: 仅若干组合逻辑门

## 六、误预测检测与恢复

### 检测条件 (`vsrc/exu/exu.v:286-287`)

```verilog
wire mispredict = (is_control && (i_predict_taken != actual_taken)) ||
                  ((i_jal || i_jalr) && i_predict_taken &&
                   ({i_predict_target, 2'b00} != o_pc_next));
```

| 类型 | 方向检查 | 目标检查 |
|------|---------|---------|
| 条件分支 (BRCH) | ✅ | ❌ |
| 无条件跳转 (JAL) | ✅ | ✅ |
| 间接跳转 (JALR) | ✅ | ✅ |

### 恢复流程

1. EXU 检测误预测 → `o_mispredict_flush = 1` (组合逻辑)
2. `exu_mispredict_flush_r`/`exu_redirect_pc_r` latch 机制确保稳定 1 周期脉冲
3. IFU 加载正确的 PC → 重定向取指
4. IFU→IDU, IDU→EXU 流水线寄存器被冲刷
5. EXU `post_valid` 被 `i_flush` 清零, 阻止错误指令进入 EXU→WBU

### 关键 Bug 修复

**Bug A: 组合信号竞争** — `exu_mispredict_flush` 在 EXU 指令切换时消失, 导致 `_r` 寄存器捕获到错误 PC。修复: latch 机制在首次检测时锁定值, 后续周期自清零。

**Bug B: JAL target 未检测** — 原代码只对 JALR 做目标匹配检测。修复: 扩展为 `(i_jal || i_jalr)`。

**Bug C: WBU pc_update 不可跳过** — 尝试对正确预测的分支跳过 `pc_update` 导致 20 个测试失败。根因: EXU→WBU 寄存器必须在每个 branch/JAL 后重置, 否则产生流水线不一致。修复: 保留 WBU 对所有控制流指令的 `pc_update`。

## 十一、pc_update 与预测器的性能冲突

### 周期级追踪 (一条 taken branch)

**原始 CPU (无预测器)** — branch at B, target T:

```
C0: IFU 取 B (分支指令)
C1: IFU 取 B+4, IDU 译码 B
C2: IFU 取 B+8, IDU 得到 B+4, EXU 执行 B → 跳转!
C3: WBU 得到 B → pc_update
    → 流水线全清, IFU 加载 T
C4: IFU 取 T
C5: IDU 得到 T
C6: T 到达 EXU              ← 第 6 周期
```

**当前 CPU (有预测器, BTB 命中)**

```
C0: IFU 取 B, BTB 命中 → 预测跳转, next_pc=T
C1: IFU 取 T ✓(正确!),  IDU 译码 B
C2: IFU 取 T+4,         IDU 得到 T,     EXU 执行 B → 正确预测
C3: IFU 取 T+8,         IDU 得到 T+4,   EXU 执行 T
    WBU 得到 B → pc_update!!!
    → EXU→WBU 清 (T 刚刚进去，清除！)
    → IDU→EXU 清 (T+4 清除！)
    → IFU→IDU 清 (T+8 清除！)
    → IFU 加载 T (和预测器目标一样，冗余)
C4: IFU 取 T (重新取！流水线空了)
C5: IDU 得到 T
C6: T 到达 EXU              ← 同样是第 6 周期！
```

### 关键发现

```
预测器的贡献:  [C1 提前取 T]  [C2 提前取 T+4]  [C3 提前取 T+8]  ← 省了 2 cycle
pc_update 的破坏: [C3 清 T]    [C3 清 T+4]      [C3 清 T+8]      ← 全部白干
────────────────────────────────────────────────────────────────
最终:          T 在 C6 到达 EXU = 和原始 CPU 完全一样
```

**预测器提前 2 周期取到了正确代码，但 pc_update 在 C3 全清了。**
T 到达 EXU 的时间 (C6) 和原始 CPU 完全相同。预测器的收益被 pc_update 完全抵消。

### 为什么 37 测试反而快了？

唯一的积极效果是 **ICache 预升温**:

```
原始:  C3 pc_update → C4 IFU 取 T → ICache miss → 停顿 N 周期
预测:  C1 IFU 取 T → ICache 已加载 → C4 再取 T → ICache HIT → 0 停顿
```

对于 quick-sort (.text=3868B, 几乎完全在 4KB ICache 内):
- 每次 JAL 调用 partition — 预测器提前把 partition 代码取进 ICache
- pc_update 清流水线后, 再取 partition → **ICache hit** → 省下访问 AXI 总线的周期
- 这就是 **47% 提速**的来源: 不是分支预测减少了惩罚, 而是减少了 ICache miss

对于 CoreMark (.text=15KB, 远超 4KB ICache):
- 预测器提前取 target, 但 target 在重取之前已被逐出 ICache
- → **ICache 预升温无效** → 省不了任何周期
- → pc_update 开销主导 → 整体更慢

### 修复方向

见 `docs/分支预测器修复方案.md` — 4 个方案从降低 pc_update 开销到完全移除 pc_update。

## 十二、已知限制

1. **BTB target 对分支不验证** — 仅检查方向, 不检查目标地址 (条件分支 target 由指令编码决定, 不会变化)
2. **RAS 仅处理 ra (x1)** — 不处理其他链接寄存器 (如 t0/x5)
3. **BTB 直接映射** — 同 index 不同 tag 的冲突会覆盖旧条目
4. **8 级 RAS 深度** — 超过 8 层嵌套的调用会丢失最旧返回地址
5. **WBU pc_update 抵消预测收益** — 预测器提前取的正确代码被 pc_update 清掉重取 (详见第十一章)
