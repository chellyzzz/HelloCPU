## 一、概述

为 hcpu 设计并实现了一套分支预测系统，包含 **BTB (Branch Target Buffer)** 和 **RAS (Return Address Stack)**，结合 **JAL 静态预测**，显著减少了控制流指令的流水线惩罚。

### 性能收益

| 测试 | 原始周期 | 预测器周期 | 提升 |
|------|---------|-----------|------|
| quick-sort | 15,458 | 8,346 | **-46%** |
| recursion | 47,638 | 13,379 | **-72%** |
| bubble-sort | 13,878 | 10,515 | **-24%** |
| select-sort | 10,565 | 8,696 | **-18%** |
| hello | 394 | 334 | **-15%** |

## 二、架构设计

```
IFU 阶段 (PC 选择)
┌─────────────────────────────────────────────────────┐
│  pc_next ─┬──→ ICache ──→ ins[31:0]                │
│           ├──→ BTB lookup ──→ predict_valid,target  │
│           ├──→ RAS lookup  ──→ ras_valid,ras_target │
│           └──→ JAL detect  ──→ jal_target (部分译码) │
│                                                     │
│  next_pc = BTB_hit  ? btb_target  :                 │
│            JAL_det ? jal_target  :                 │
│            RAS_hit ? ras_target  :                 │
│            pc+4                                     │
└─────────────────────────────────────────────────────┘

EXU 阶段 (验证/恢复)
┌──────────────────────────────────────────────────────┐
│  实际分支结果 vs predict_taken → mispredict?          │
│  是 → flush IFU/IDU, 重定向 PC, 更新 BTB/RAS         │
│  否 → 更新 BTB 计数器                                │
└──────────────────────────────────────────────────────┘
```

## 三、BTB — 分支目标缓冲

### 结构

- **条目数**: 64
- **映射方式**: 直接映射 (Direct-mapped)
- **索引**: `pc[7:2]` (6-bit, 每条 4 字节指令一个槽)
- **Tag**: `pc[31:8]` (24-bit)

### 条目格式

```
{ valid[1], tag[24], target[30], counter[2] }
  有效位    PC[31:8]  地址[31:2]   2-bit 饱和计数器
                                 (00=SN,01=WN,10=WT,11=ST)
```

预测跳转条件: `counter[1] == 1`

### 行为

| 场景 | BTB 操作 |
|------|---------|
| **Lookup** | `pc[7:2]` 索引 → tag 比对 → `counter[1]` 决定预测方向 |
| **分支跳转 (update)** | 分配/更新条目, counter → ST, 更新 target |
| **分支不跳 (update)** | counter → SN, 不预测跳转 |
| **首次遇到分支** | BTB miss → 预测不跳, 若实际跳转则下次分配条目 |

### 源文件

`vsrc/ifu/btb.v` — 纯时钟同步模块, 组合逻辑 lookup + 时钟边沿 update

---

## 四、RAS — 返回地址栈

### 结构

- **深度**: 8 条目, LIFO 栈
- **条目宽度**: 30-bit (地址[31:2])

### 操作规则

| 操作 | 检测条件 (EXU) | 时机 |
|------|------------|------|
| **Push** | `JAL` 且 `rd == x1(ra)` | EXU 提交阶段 |
| **Pop** | `JALR` 且 `rd == x0` 且 `rs1 == x1(ra)` | EXU 提交阶段 |
| **Predict** | `JALR` 且 `rd == x0` (IFU 阶段) | IFU 取指阶段 |

预测时仅读取栈顶 (不弹出) → 弹出在 EXU 阶段执行。

### 注意事项

- RAS 更新在 EXU 阶段 (非推测执行), 避免误预测时状态损坏
- 若栈空时发生 Return → RAS miss → 依靠 EXU mispredict 重定向
- 若嵌套深度超过 8 → 最旧条目被丢弃 (不崩溃)

### 源文件

`vsrc/ifu/ras.v` — 纯时钟同步模块, 组合逻辑读栈顶 + 时钟边沿 push/pop

## 五、JAL 静态预测

### 原理

JAL 指令总是跳转。在 IFU 阶段对 `icache_ins` 做最少量的部分译码:

```verilog
wire is_jal  = (icache_ins[6:0] == 7'b1101111);
wire [31:0] jal_imm = {{12{icache_ins[31]}}, icache_ins[19:12],
                        icache_ins[20], icache_ins[30:21], 1'b0};
wire [31:0] jal_target = pc_next + jal_imm;
```

J-type immediate 提取与 `vsrc/idu/idu.v:106` 完全一致, 经验证 0 次 target mismatch。

### 收益

- JAL 指令 0 周期惩罚 (vs 原始 2-3 周期)
- 无需 BTB 条目存储
- 面积开销: 仅若干组合逻辑门

## 六、误预测检测与恢复

### 检测条件 (`vsrc/exu/exu.v:286-287`)

```verilog
wire actual_taken = (i_brch && brch_res) || i_jal || i_jalr;
wire is_control   = i_brch || i_jal || i_jalr;

wire mispredict = (is_control && (i_predict_taken != actual_taken)) ||
                  ((i_jal || i_jalr || i_brch) && i_predict_taken &&
                   ({i_predict_target, 2'b00} != o_pc_next));
```

| 类型 | 方向检查 | 目标检查 |
|------|---------|---------|
| 条件分支 (BRCH) | ✅ | ❌ |
| 无条件跳转 (JAL) | ✅ | ✅ |
| 间接跳转 (JALR) | ✅ | ✅ |

### 恢复流程

1. EXU 检测误预测 → `o_mispredict_flush = 1` (组合逻辑)
2. 误预测 latch 机制 (`mispredict_latched` FSM) 确保 `exu_mispredict_flush_r` 为稳定的 1 周期脉冲
3. IFU 优先响应 `exu_mispredict_flush_r` → 加载 `exu_redirect_pc_r` 重定向取指
4. IFU→IDU、IDU→EXU 流水线寄存器被 `exu_mispredict_flush_r` 冲刷
5. EXU→WBU 在后续周期被 WBU 的 `pc_update_en` 冲刷

### 误预测 Latch 机制 (`vsrc/top/hcpu.v:529-544`)

```verilog
reg mispredict_latched;
always @(posedge clock or posedge reset) begin
    if (reset) begin
        exu_mispredict_flush_r <= 1'b0;
        exu_redirect_pc_r      <= 32'b0;
        mispredict_latched     <= 1'b0;
    end else begin
        if (exu_mispredict_flush && !mispredict_latched) begin
            exu_redirect_pc_r  <= exu_redirect_pc;
            mispredict_latched <= 1'b1;
        end
        exu_mispredict_flush_r <= exu_mispredict_flush && !mispredict_latched;
        if (!exu_mispredict_flush)
            mispredict_latched <= 1'b0;
    end
end
```

**时序**：
- Cycle T: EXU 检测误预测, `exu_mispredict_flush = 1`
- posedge T: latch 捕获 `exu_redirect_pc_r`, 设 `mispredict_latched = 1`, `exu_mispredict_flush_r = 1`
- Cycle T+1: `exu_mispredict_flush_r = 1` → IFU 重定向, 流水线冲刷
- posedge T+1: 若 `exu_mispredict_flush = 0`, 清零 `mispredict_latched` 和 `exu_mispredict_flush_r`

## 七、EXU→WBU Gating Fix — 防止错误路径污染

### 问题

误预测检测后，IDU→EXU 被 `exu_mispredict_flush_r` 冲刷，但 EXU→WBU 仅受 `pc_update_en` 重置。
`pc_update_en` 来自 WBU FSM 的 `o_pc_update`，比 `exu_mispredict_flush_r` 晚 **1 拍**。

这个 1 拍的空窗期让错误路径指令（在误预测分支后的指令）有机会：
1. 进入 IDU→EXU（在 `exu_mispredict_flush_r` 生效前的 posedge）
2. 在 EXU 中进行组合逻辑执行
3. 被锁存进 EXU→WBU（此时 `exu_mispredict_flush_r` 已生效，但 EXU→WBU reset 不含该信号）
4. 在 WBU 阶段写入寄存器文件 → **寄存器状态被污染**

### 修复

添加 `i_flush` 输入端口到 `exu_wbu_regs`，连接到 `exu_mispredict_flush_r`。
在 latch 条件中增加门控：`&& !i_flush`，在 flush 信号有效的周期阻止锁存。

```verilog
// vsrc/exu/exu_wbu_regs.v — latch condition change
else if(i_post_ready && o_post_valid && !i_flush) begin
    // latch data normally
end

// vsrc/top/hcpu.v — connection
.i_flush (exu_mispredict_flush_r)
```

### 为什么是门控（gating）而非复位（reset）？

| 方法 | 周期 T 行为 | WBU 可见性 | 风险 |
|------|------------|-----------|------|
| **Reset** (`|| flush_r`) | EXU→WBU 全部清零 | 分支自身数据被清除（WBU 从 i_brch 读回 0） | 竞态：WBU FSM 可能未触发 pc_update |
| **Gating** (`&& !i_flush`) | 不锁存新数据，但保留现有输出 | 分支数据保留（WBU 正确看到 i_brch=1） | ✅ 无竞态 |

用复位的方式存在死循环风险：WBU FSM 可能因读取被复位的 i_brch=0 而跳过 pc_update，导致流水线未正确冲刷。

### 效果

40/40 cpu-tests 全部通过（含 3 个 BTB 专项测试）。

## 十一、CoreMark 已知问题

### 实验矩阵

| 预测组合 | CoreMark ITER=1 | 说明 |
|----------|----------------|------|
| 全关 (无预测) | ✅ CRC 正确, 856K cycles | 基线 |
| BTB 独占 (JAL/RAS 关) | ❌ CRC 错误 (crclist 0xc577) | 误预测冲刷后仍可能有少量错误路径污染 |
| JAL 独占 (BTB/RAS 关) | ❌ 死循环 (45M+ inst 后不终止) | JAL 预测改变流水线时序 |
| BTB+JAL (RAS 关) | ❌ 死循环 (80M+ inst) | 同上 |
| 全开 (BTB+JAL+RAS) | ❌ 死循环 (80M+ inst) | 同上 |

### 死循环分析

JAL 预测导致死循环的根本原因尚未确定。理论分析表明 JAL 预测的时序应该是正确的：

- JAL target 由指令立即数和 PC 组合逻辑计算，与 IDU→EXU 中的立即数完全一致
- 转发路径保证 ra (返回地址) 在调用者读取时已写回
- `i_flush` 门控防止错误路径指令污染寄存器文件

可能原因：
1. **pc_update 多周期冲刷窗口** — 误预测冲刷 (1 拍) + pc_update 冲刷 (1 拍) 合计 2 拍，可能形成连锁冲刷循环
2. **JAL 预测提前取指的副作用** — 目标代码提前进入流水线，被后续冲刷清除后重新取指，某些极端模式可能导致清理不完全

### CRC 错误分析 (BTB-only)

BTB 误预测导致错误路径指令可能在特定时序条件下绕过 `i_flush` 门控进入 WBU。需要进一步完善冲刷机制或增加锁步验证。

### 下一步

1. 添加死循环检测器 — DPI-C 函数监控 PC 值在小范围内循环超过 N 次后打印诊断信息
2. 锁步对比 — 逐指令比较无预测 vs 有预测的寄存器文件状态，定位第一条分歧指令
3. 基于根因选择修复方案（参见 `docs/分支预测器修复方案.md`）

### 关于 wb_reg 解耦方案的教训

尝试将 WBU 拆分为数据旁路 (continuous assignment) + PC 控制 (FSM) 的方案 (`vsrc/wbu/wb_reg.v`) 失败。根因：

```
RegisterFile 转发端口:
  .wbu_rd    → exu2wbu_rd_addr   (实时，覆盖 EXU→WBU 当前指令)
  .waddr     → wbu_rd_addr       (已通过 wb_reg 延迟，落后 1 拍)

结果: T+2 周期时 exu2wbu_* 已移至下条指令，
      但 rf[] 尚未写入 (wb_reg 还在延迟中) → 无转发源 → 数据丢失
```

结论：写回路径不能做任何额外延迟——转发网络需要数据的实时可用性。

## 十三、pc_update 与预测器的性能冲突

### 周期级追踪 (一条 taken branch)

**原始 CPU (无预测器)** — branch at B, target T:

```
C0: IFU 取 B (分支指令)
C1: IFU 取 B+4, IDU 译码 B
C2: IFU 取 B+8, IDU 得到 B+4, EXU 执行 B → 跳转!
C3: WBU 得到 B → pc_update
    → 流水线全清, IFU 加载 T
C4: IFU 取 T
C5: IDU 得到 T
C6: T 到达 EXU              ← 第 6 周期
```

**当前 CPU (有预测器, 正确预测)**

```
C0: IFU 取 B, 预测跳转, next_pc=T
C1: IFU 取 T ✓,  IDU 译码 B
C2: IFU 取 T+4, IDU 得到 T,    EXU 执行 B → 正确预测
C3: IFU 取 T+8, IDU 得到 T+4,  EXU 执行 T
    WBU 得到 B → pc_update!!!
    → EXU→WBU 清 (T 刚刚进去，清除！)
    → IDU→EXU 清 (T+4 清除！)
    → IFU→IDU 清 (T+8 清除！)
    → IFU 加载 T (冗余)
C4: IFU 取 T (重新取！流水线空了)
C5: IDU 得到 T
C6: T 到达 EXU              ← 同样是第 6 周期！
```

### 关键发现

```
预测器的贡献:  [C1 提前取 T]  [C2 提前取 T+4]  [C3 提前取 T+8]  ← 省了 2 cycle
pc_update 的破坏: [C3 清 T]    [C3 清 T+4]      [C3 清 T+8]      ← 全部白干
────────────────────────────────────────────────────────────────
最终:          T 在 C6 到达 EXU = 和原始 CPU 完全一样
```

**预测器提前 2 周期取到了正确代码，但 pc_update 在 C3 全清了。**
T 到达 EXU 的时间 (C6) 和原始 CPU 完全相同。预测器的收益被 pc_update 完全抵消。

### 为什么 37 测试反而快了？

唯一的积极效果是 **ICache 预升温**:

```
原始:  C3 pc_update → C4 IFU 取 T → ICache miss → 停顿 N 周期
预测:  C1 IFU 取 T → ICache 已加载 → C4 再取 T → ICache HIT → 0 停顿
```

对于 quick-sort (.text=3868B, 几乎完全在 4KB ICache 内):
- 每次 JAL 调用 partition — 预测器提前把 partition 代码取进 ICache
- pc_update 清流水线后, 再取 partition → **ICache hit** → 省下访问 AXI 总线的周期
- 这就是 **47% 提速**的来源: 不是分支预测减少了惩罚, 而是减少了 ICache miss

对于 CoreMark (.text=15KB, 远超 4KB ICache):
- 预测器提前取 target, 但 target 在重取之前已被逐出 ICache
- → **ICache 预升温无效** → 省不了任何周期
- → pc_update 开销主导 → 整体更慢

### 修复方向

见 `docs/分支预测器修复方案.md` — 多个方案从降低 pc_update 开销到完全移除 pc_update。

## 十四、已知限制

1. **BTB target 对分支不验证** — 仅检查方向, 不检查目标地址 (条件分支 target 由指令编码决定, 不会变化)
2. **RAS 仅处理 ra (x1)** — 不处理其他链接寄存器 (如 t0/x5)
3. **BTB 直接映射** — 同 index 不同 tag 的冲突会覆盖旧条目
4. **8 级 RAS 深度** — 超过 8 层嵌套的调用会丢失最旧返回地址
5. **WBU pc_update 抵消预测收益** — 预测器提前取的正确代码被 pc_update 清掉重取 (详见第十三章)
6. **CoreMark 死循环** — JAL 预测在 CoreMark 上导致死循环 (详见第十一章)
7. **CoreMark CRC 错误** — BTB-only 模式 CRC 校验失败 (详见第十一章)
