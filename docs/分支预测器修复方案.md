# 分支预测器修复与改良方案

> 创建日期: 2026-05-03 | 状态: 分析中

---

## 一、问题陈述

### 现象

| 配置 | 37 测试 | CoreMark ITER=1 |
|------|--------|-----------------|
| 原始 CPU (无预测器) | 37/37 PASS | ~1.8M cycles 完成 |
| BTB+JAL+RAS (当前) | 37/37 PASS | ~15M cycles 未完成 (8x 开销) |
| 全预测 OFF (当前结构) | 37/37 PASS | ~20M cycles 未完成 (更慢) |

37 测试中有显著加速 (quick-sort 47%, recursion 72%)，但 CoreMark 反而变慢。

### 根因

WBU `pc_update` 对每条 taken branch 触发完整流水线冲刷:

```
C0: IFU 取 branch B, 预测跳转, 提前取 target T
C1: IFU 取 T+4, IDU 得到 T
C2: IFU 取 T+8, IDU 得到 T+4, EXU 执行 B → 预测正确
C3: WBU 得到 B → pc_update!
    → EXU→WBU 清 (T 刚刚进去，清除！)
    → IDU→EXU 清 (T+4 清除！)
    → IFU→IDU 清 (T+8 清除！)
    → IFU 重定向到 T (冗余，预测器已经在了)
C4: IFU 重新取 T (流水线全空)
```

预测器在 C1 就提前取到了正确的 target 代码，但 C3 的 pc_update 把它们全部清掉。**预测器节省的周期被 pc_update 抵消了。**

### 为什么 pc_update 不能跳过？

实验记录：

| 实验 | pc_update 范围 | 37 测试 | CoreMark |
|------|---------------|---------|----------|
| 原始 (跳过全部) | 仅 JALR+ECALL+MRET | 17/37 FAIL | TIMEOUT |
| 恢复全部 | 所有控制流 | **37/37 PASS** | 慢但正常 |
| 仅 EXU→WBU 清 (留 IFU→IDU/IDU→EXU) | 所有控制流 | CoreMark 死循环 | — |
| 跳过预测 JAL | JAL 跳过, branch 保留 | **35/37 FAIL** | — |
| 跳过预测 branch | branch 跳过, JAL 保留 | 未测 | — |

## 二、为什么 EXU→WBU 必须被清？

### 完整周期追踪

```
分支 B 在 EXU→WBU 时:
  周期0: EXU 执行 B, 产生 o_res, o_pc_next, o_brch
  posedge0: EXU→WBU 锁存 B 的数据
          → EXU→WBU 输出: o_brch=1, o_pc_next=target

  周期1: WBU 看到 B 的数据, 触发 pc_update
  posedge1: EXU→WBU 锁存 B+1 的数据 (来自 EXU)
          Delta: pc_update_en=1 → EXU→WBU RESET
          → B+1 的数据刚锁存就被清除了！

  周期2: WBU self-clear pc_update
  posedge2: B+1 重新取指 → 进入流水线
```

**如果没有 pc_update RESET EXU→WBU:**

```
  posedge1: EXU→WBU 锁存 B+1 的数据 (未被清)
  周期2: WBU 看到 B+1 的数据 (在 pc_update self-clear 的同一周期)
  posedge2: WBU 锁存 B+1
```

理论上 B+1 的数据是正确的（预测器提前取的 target 代码），不应被清。**但实验结果（20 个测试失败）表明这样做会导致错误**。

### 假设：指令重排序

当 EXU→WBU 被清但 IFU→IDU 和 IDU→EXU 不清时:

```
清除前:  EXU→WBU[B+1]  IDU→EXU[B+2]  IFU→IDU[B+3]  IFU→target
清除后:  EXU→WBU[空]   IDU→EXU[B+2]  IFU→IDU[B+3]  IFU→target

结果:    B+2 在 B+1 之前执行 → 指令重排序 → 数据依赖错误
```

**全清或全不清是正确的，部分清除会产生指令重排序。**

### 关键问题

**全不清 (跳过 pc_update) 为什么也失败？**

当所有流水线阶段都不清时，指令顺序正确 (B+1, B+2, B+3)，不应有依赖错误。但实验显示 20 个测试失败。可能原因:

1. **EXU→WBU 的数据在 pc_update 周期被 WBU 错误处理** — WBU 在 self-clear 状态时可能干扰 B+1 的数据
2. **EXU→WBU 的 handshake 与 WBU pc_update 冲突** — WBU 在处理分支的状态转换时可能错误接受/拒绝 B+1
3. **delta-cycle 竞争** — pc_update_en 的边沿可能意外触发某些模块的重置

**推荐做法：用 GTKWave 追踪一个失败测试（bubble-sort, 1108 cycles）中第一条被错误执行的指令，确认具体是哪种情况。**

## 三、修复方案

### 方案 A: 将 pc_update 延迟到分支完全离开 WBU 后 (推荐)

**思路**: 产生一个新的 `commit_flush` 信号，在分支离开 WBU **之后**才清 EXU→WBU，而不是在分支刚进入 WBU 时。

```
原始时序:
  posedge0: B 进入 EXU→WBU
  posedge1: B 进入 WBU, B+1 进入 EXU→WBU, pc_update 触发
           → B+1 在 EXU→WBU 刚锁存就被清除

修复时序:
  posedge0: B 进入 EXU→WBU
  posedge1: B 进入 WBU, B+1 进入 EXU→WBU
           → 不触发 flush
  posedge2: B 离开 WBU (self-clear), B+1 进入 WBU
           → commit_flush 触发 (1 cycle 脉冲)
           → 清 EXU→WBU (此时 B+1 已进入 WBU, 清的是 B+2)
```

**关键改变**: `commit_flush` 在分支离开 WBU 后发出，此时:
- B+1 已安全进入 WBU
- B+2 在 EXU→WBU 被清
- B+2 重新取指 (和原始 pc_update 行为一致)

**实现**:
```verilog
// WBU: 新增 3 状态 FSM
// S_IDLE → S_PCUPDATE → S_COMMIT_FLUSH → S_IDLE

reg [1:0] state;
always @(posedge clock) begin
    case (state)
        S_IDLE:
            if (i_jal || i_jalr || i_brch || i_ecall || i_mret) begin
                o_pc_next <= i_pc_next;
                o_pc_update <= (i_jalr && ~i_predict_taken) || i_ecall || i_mret;
                state <= S_PCUPDATE;
            end
        S_PCUPDATE:
            state <= S_COMMIT_FLUSH;  // o_commit_flush <= 1
        S_COMMIT_FLUSH:
            state <= S_IDLE;           // o_commit_flush <= 0
    endcase
end
```

`o_commit_flush` 在 S_COMMIT_FLUSH 周期为 1，连接至:
- EXU→WBU reset
- IFU→IDU reset (仅对 branch，防止部分清除)
- IDU→EXU reset (同上)

**优点**: 时序正确，B+1 安全到达 WBU
**缺点**: 增加 3 周期 WBU 状态机，每个分支额外 1 周期延迟

### 方案 B: 在 EXU 阶段直接做目标地址比对 + 跳过正确预测的 pc_update

**思路**: 不改变 pc_update 机制，而是在 EXU 阶段做额外的目标地址验证。当分支正确预测时，产生一个 `pred_correct` 信号锁存到 EXU→WBU，WBU 看到该信号后跳过 pc_update。

**关键问题**: 方案 A 分析中提到的"为什么全不清也失败"必须先回答。如果在 GTKWave 中确认全不清是 WBU handshake 冲突导致的，则此方案只需要修复 WBU handshake。

**实现**:
```verilog
// EXU: 锁存预测正确标志
reg pred_correct;
assign o_pred_correct = pred_correct && (i_predict_taken == actual_taken) 
                         && (!i_jal || ({i_predict_target,2'b00} == o_pc_next));

// EXU→WBU: 新增 o_pred_correct 传递
// WBU: 当 i_pred_correct=1 时跳过 pc_update
```

**优点**: 最小的修改量
**缺点**: 需先确认根因

### 方案 C: 完全移除 pc_update，用专用的 commit flush 替代

**思路**: 完全抛弃 pc_update 机制，替换为:
1. `exu_mispredict_flush` → 处理误预测 (已有)
2. `wbu_commit_flush` → 处理分支提交后的流水线清理 (新增)

`wbu_commit_flush` 在分支离开 WBU 后触发，清 EXU→WBU，但不清 IFU→IDU 和 IDU→EXU (因为预测器已正确取指)。

**与方案 A 的区别**: 方案 C 不清 IFU→IDU 和 IDU→EXU，仅清 EXU→WBU。

**优点**: 最彻底，预测器收益最大化
**缺点**: 需要解决"全不清"时的指令重排序问题

### 方案 D: 不做任何改变，接受当前性能

**思路**: 37/37 测试通过，CoreMark 虽然在当前超时设置下未完成但正在正确执行。增加 `max_cycles` 或优化其他瓶颈。

**优点**: 零开发成本
**缺点**: CoreMark 8x 慢于原始 CPU

## 四、建议实验顺序

### 阶段 1: 用 GTKWave 确认根因 (2-4 小时)

```bash
make sim
./build/Vsim_top sw/build/bubble-sort.bin --wave --max-cycles=2000
gtkwave wave.vcd
```

在 GTKWave 中追踪:
- `pc_update_en` (WBU 输出)
- `exu2wbu_valid` (EXU→WBU valid)
- `exu2wbu_wen` (EXU→WBU write enable)
- `wbu2exu_ready` (WBU→EXU ready)
- `wbu_wen` (WBU write enable)
- `exu_mispredict_flush_r` (误预测 flush)

定位第一个被错误执行/写入的指令及其 PC。

### 阶段 2: 根据根因选择方案

- 如果根因是 WBU handshake 竞态 → 方案 A 或 B
- 如果根因是指令重排序 → 方案 C
- 如果根因是 delta-cycle 问题 → 方案 A

### 阶段 3: 实现并测试

1. 实现所选方案的 RTL 修改
2. `make sim && make run` 验证 37/37
3. `make bench_only ITER=1` 验证 CoreMark 不卡死
4. `make bench_only ITER=100` 跑标准跑分

|------|--------|------------------------------|---------|-----------|
| 当前 | 37/37 | ~15M | 8x 慢 | — |
