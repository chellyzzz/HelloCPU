## 二、已完成修复

### Fix 1: EXU→WBU Gating (✅ 已提交 `64b8ee2`)

**问题**: 误预测冲刷后，EXU→WBU 的 reset (`pc_update_en`) 比冲刷信号 (`exu_mispredict_flush_r`) 晚 1 拍，导致错误路径指令有 1 拍空窗期被锁存进 WBU。

**修复**: 在 `exu_wbu_regs` latch 条件中增加 `!i_flush` 门控。

```verilog
// vsrc/exu/exu_wbu_regs.v — latch gating
else if(i_post_ready && o_post_valid && !i_flush) begin
    ...
end

// vsrc/top/hcpu.v — 连接
.i_flush (exu_mispredict_flush_r)
```

**为什么不是 reset 而是 gating**: 若用 reset (`|| exu_mispredict_flush_r`)，会在 posedge 同时将 EXU→WBU 清零和 WBU FSM 评估，可能导致竞态（WBU FSM 看到 i_brch=0 而跳过 pc_update）。Gating 方案保留分支自身数据，仅阻止新数据锁存。

**效果**: 40/40 cpu-tests PASS。

### 已排出的死路

#### wb_reg 解耦方案

尝试将 WBU 拆分为数据旁路 + PC 控制 FSM (`vsrc/wbu/wb_reg.v` → 额外一级写回寄存器)。失败。

**失败原因**:
```
RegisterFile 转发端口:
  .wbu_rd    → exu2wbu_rd_addr   (实时)
  .waddr     → wbu_rd_addr       (已通过 wb_reg 延迟 1 拍)

T+2 周期: exu2wbu_* 已移至下条指令
           rf[] 尚未写入
           转发源缺失 → 数据丢失 → 几乎所有 cpu-tests FAIL
```

**教训**: 转发网络需要数据实时可用——写回路径不能增加任何延迟。

## 三、CoreMark 死循环机制分析

### 完整实验矩阵

| 日期 | BTB | JAL | RAS | cpu-tests | CoreMark | 备注 |
|------|-----|-----|-----|-----------|----------|------|
| 05-03 | ON | ON | ON | 37/37 | 死循环 | 初始版本 |
| 05-04 | ON | ON | ON | 40/40 | 死循环 | +gating fix |
| 05-04 | ON | ON | OFF | 40/40 | 死循环 | RAS 无关 |
| 05-04 | OFF | ON | OFF | 40/40 | 死循环 | BTB 无关, 确认为 JAL |
| 05-04 | ON | OFF | OFF | 40/40 | CRC 错误 | 完成但数据错 |
| 05-04 | OFF | OFF | OFF | 40/40 | ✅ 正确 | 基线 |

结论：**JAL 预测是死循环的充要条件**（只要 JAL 预测开启就死循环），BTB 和 RAS 无关。

### 理论分析：JAL 预测为什么应该工作

JAL 预测（`pred_taken_jal = is_jal`）：

1. IFU 从指令位抽取 J-immediate，计算 `jal_target = pc_next + jal_imm`
2. 立即数编码与 IDU 完全一致 (`idu.v:106`)
3. EXU 中做 target 验证：`{i_predict_target, 2'b00} != o_pc_next`
4. 所有 37 个 cpu-tests 中 **0 次 JAL target mismatch**（`jal_tgt_bad: 0`）
5. 转发路径保证 `ra = pc+4` 在调用者读取前已写回

**理论上 JAL 预测 100% 正确**，不应产生任何误预测。但 CoreMark 仍然死循环。

### 假设：pc_update 多周期冲刷连锁

对于正确预测的 JAL，流水线在 WBU 阶段触发 `pc_update`。冲刷周期为：

```
posedge T:   JAL 进入 EXU→WBU
Cycle T+1:   EXU→WBU output = JAL data, WBU FSM 置 o_pc_update
posedge T+1: pc_update=1 → IFU→IDU、IDU→EXU、EXU→WBU 全清
             但目标指令已提前进入 IFU→IDU (T-1 就取指了)
posedge T+1: 目标指令被窜冲, IFU 重定向到同一 target (冗余)
Cycle T+2:   IFU 重新取目标指令
```

每个正确预测的 JAL 产生 **1 个冗余冲刷周期**。如果 JAL 频繁出现（函数调用密集），冲刷窗口可能叠加，形成连锁效应。

但如果函数最终返回而且循环计数器正确递减，程序应该仍然终止。死循环说明某些指令（可能是循环计数器递减 `addi`）被冲刷丢失后未能重新执行。

### 下一步调试方向

1. **锁步对比** — 在 sim_main.cpp 中逐指令对比无预测 vs JAL 预测的寄存器文件状态
2. **死循环检测器** — DPI-C 函数监控 PC 在小范围内重复超过 N 次
3. **周期级追踪** — 在死循环中的单次迭代做 VCD 波形对比

## 四、CoreMark CRC 错误机制分析 (BTB-only)

### 现象

BTB 预测产生 CRC 错误：`crclist 0xc577 (≠ 0xe714)`, `crcstate 0x3310 (≠ 0x8e3a)`。

### 根因分析

BTB 预测错误时：

1. 分支在 EXU 检测到误预测
2. `exu_mispredict_flush_r` → IFU→IDU 和 IDU→EXU 冲刷
3. `i_flush` 门控阻止错误路径指令进入 EXU→WBU
4. 但 **正确路径指令可能也被延迟**（多周期冲刷）

潜在的残余污染：

- **分支目标计算错误**：BTB 直接映射导致别名（aliasing），一个 BTB 条目可能被不同分支共享。分支 A 的更新可能覆盖分支 B 的条目，导致 B 用错误的 target 预测。虽然 target 错误会在 EXU 检测出来（mispredict），但冲刷过程中的指令可能已造成破坏。
- **冲刷链中的指令**：两段冲刷（mispredict_flush + pc_update）之间的指令可能已在寄存器文件中写入数据。

### 验证思路

1. 对比 BTB-only 和无预测的寄存器文件终态
2. 在 BTB update 时记录（PC, target, taken），追踪冲突模式

## 六、历史实验记录

### 跳过 pc_update 实验

| 实验 | 描述 | 结果 |
|------|------|------|
| 跳过全部预测 | JALR+ECALL+MRET 仅触发 pc_update | 17/37 FAIL |
| 恢复全部 | 所有控制流触发 pc_update | **37/37 PASS** |
| 仅清 EXU→WBU | 保留 IFU→IDU/IDU→EXU | CoreMark 死循环 |

| 跳过正确预测 branch | 带 `i_pred_correct` 信号 | 未测试 |

### EXU→WBU 复位 vs 门控

| 实验 | 描述 | 结果 |
|------|------|------|
| 标准 (无 gating) | `reset = reset \|\| pc_update_en` | 37/37 PASS, CoreMark CRC 错误 |
| 复位方式 | `reset = ...\|\| exu_mispredict_flush_r` | cpu-tests 超时 (过于激进) |
| **门控方式** | `&& !i_flush` 门控 latch | **40/40 PASS** (✅ 当前方案) |
