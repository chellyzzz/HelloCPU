# HelloCPU 向量协处理器接口草案

## 一、文档目标

本文档给出 HelloCPU 与未来向量协处理器之间的一版**粗接口草案**。

目标不是一次性把所有细节定死，而是先统一以下内容：

1. CPU 与向量协处理器之间的边界；
2. issue / complete / commit / kill 的基本语义；
3. 允许 CPU 侧和向量侧并行开发的最小接口集合；
4. 后续细化时尽量不推翻当前框架。

本文档强调的是 **接口语义先行，具体实现后定**。

---

## 二、设计原则

### 1. 协处理器是“执行后端”，不是特殊函数调用

向量协处理器在系统中的定位应当更接近：

- 一个新的 issue target；
- 一个新的长延迟结果来源；
- 一个新的异常来源；
- 未来可能具备独立访存能力的后端单元。

不建议把它设计成 EXU 里的一堆特殊 if/else 路径，否则后续在：

- flush；
- trap；
- 异常恢复；
- 访存仲裁；

这些问题上会迅速失控。

### 2. 先冻结接口语义，不冻结内部实现

当前阶段先定义：

- 请求长什么样；
- 返回长什么样；
- 什么情况下 kill；
- 什么情况下允许提交；
- 异常如何上报。

而不急着定义：

- 向量寄存器文件具体结构；
- lane 数量；
- 向量长度编码；
- 精确到 RTL 级的内部流水实现。

### 3. 控制流恢复语义必须统一

无论来自：

- 标量分支误预测；
- ecall / mret / trap；
- 协处理器异常；
- 向量访存异常；

都应尽量走统一的 kill / flush / redirect / commit 语义。

### 4. 第一版接口优先支持单发射顺序核

HelloCPU 当前是单发射顺序核，因此第一版接口默认：

- CPU 每拍最多发出一条协处理器请求；
- 协处理器可以多拍完成；
- CPU 保留顺序提交语义；
- 不要求第一版就支持乱序返回后直接乱序提交。

但接口字段中应预留 tag / sequence 信息，避免以后重构。

---

## 三、系统角色划分

### CPU 负责

- 指令取值、解码、标量寄存器读写；
- 指令是否发往协处理器的判定；
- 发射时携带源操作数和控制信息；
- 统一管理 flush / redirect / kill；
- 统一决定什么时候可以架构提交；
- 接收协处理器结果并写回；
- 接收协处理器异常并转入 trap 处理。

### 协处理器负责

- 接收 issue 请求；
- 在内部完成向量运算；
- 在需要时发起或请求访存；
- 返回结果或完成状态；
- 报告异常；
- 在收到 kill 后丢弃被取消的未完成操作。

### 暂不在第一版定死的边界

- 向量访存是否独立穿过 LSU；
- 向量 CSR 是否完全采用标准 RISC-V V 规范；
- 是否支持多个未完成向量操作同时在飞。

---

## 四、建议的最小接口集合

第一版建议至少定义以下接口。

### 1. Issue 请求通道

采用 valid/ready 握手：

- `cop_issue_valid`
- `cop_issue_ready`
- `cop_issue_payload`

语义：

- CPU 在解码/发射阶段确认该指令由协处理器处理时，拉高 `cop_issue_valid`；
- 协处理器可接收请求时拉高 `cop_issue_ready`；
- `valid && ready` 的那个拍，表示该请求正式进入协处理器。

### 2. Response 返回通道

采用 valid/ready 握手：

- `cop_resp_valid`
- `cop_resp_ready`
- `cop_resp_payload`

语义：

- 协处理器在某条请求完成、可回传结果或完成状态时拉高 `cop_resp_valid`；
- CPU 在可以接收返回结果时拉高 `cop_resp_ready`；
- `valid && ready` 的那个拍，表示返回被 CPU 接收。

### 3. Kill / Flush 控制通道

- `cop_kill_valid`
- `cop_kill_seq`
- `cop_kill_all`

语义：

- 当 CPU 因误预测、异常、trap、mret 等事件取消未提交指令时，向协处理器发送 kill；
- `cop_kill_all` 表示当前所有未提交向量操作全部失效；
- `cop_kill_seq` 表示只 kill 某个序号之后或某个标签对应的操作，具体编码可在细化版确定。

第一版实现可以先只支持粗粒度的 `kill_all`。

### 4. Busy / Status 状态接口

- `cop_busy`
- `cop_idle`
- `cop_fault_pending`

语义：

- `cop_busy` 表示协处理器内部仍有未完成工作；
- `cop_idle` 可视为 `!cop_busy`，是否单独保留实现可后定；
- `cop_fault_pending` 表示已有异常待 CPU 接收。

### 5. Exception 上报接口

- `cop_exc_valid`
- `cop_exc_cause`
- `cop_exc_tval`
- `cop_exc_seq`

语义：

- 协处理器在发现需要架构可见的异常时，通过该通道报告；
- CPU 负责把异常纳入统一 trap 流程；
- `seq` 用于标识异常对应的在飞操作。

---

## 五、建议的 payload 字段

### 1. `cop_issue_payload`

第一版建议至少包含：

- `opcode`：向量操作类型或子命令；
- `funct`：更细粒度的子操作编码；
- `src0_value`：标量源操作数 0；
- `src1_value`：标量源操作数 1；
- `src2_value`：可选第三源操作数；
- `dst_type`：结果写回目标类型；
- `dst_idx`：目标寄存器编号或目标槽位；
- `seq`：该请求的顺序号或 tag；
- `pc`：原始指令 PC，便于调试和异常定位；
- `instr`：原始指令编码，便于早期 bring-up；
- `rm_or_mode`：模式位，保留给舍入、掩码、长度等扩展字段；
- `flags`：是否写回、是否需要提交确认、是否可能访存等控制位。

第一版不强制区分“向量寄存器源操作数”和“标量立即数”的最终编码方式，但要预留统一字段。

### 2. `cop_resp_payload`

第一版建议至少包含：

- `seq`：返回对应哪条请求；
- `dst_type`：结果写回目标类型；
- `dst_idx`：目标编号；
- `result0`：主结果数据；
- `result1`：可选附加结果；
- `wb_enable`：是否需要 CPU 执行写回；
- `done`：该请求是否彻底完成；
- `flags`：饱和、比较结果、部分完成等状态位；
- `exc`：是否同时携带异常。

第一版可以只真正使用 `result0`，其他字段先预留。

## 六、CPU 侧控制语义建议

### 1. Issue 语义

- CPU 只有在该指令尚未被 flush 且协处理器 ready 时，才允许发射；
- 发射成功后，该操作进入“在飞”状态；
- CPU 需要记录该操作是否尚未完成、是否尚未提交、是否已被 kill。

### 2. Complete 语义

- 协处理器返回 `resp` 只表示“执行结果已经准备好”；
- 不必等同于“已经架构提交”；
- CPU 可先缓存结果，再在满足顺序提交条件时正式写回。

这一区分非常重要，因为未来需要统一：

- 标量多周期执行；
- 向量长延迟执行；
- trap / mispredict 下的 kill；
- 精确异常。

### 3. Commit 语义

- 第一版仍建议由 CPU 统一提交；
- 协处理器不直接修改架构可见状态；
- 任何写回、CSR 影响、异常生效，都应由 CPU 在 commit 点统一决定。

### 4. Kill 语义

- kill 作用于“已发射但尚未提交”的向量操作；
- 被 kill 的操作即使内部算完，也不应再产生架构可见写回；
- 若返回与 kill 同时竞争，以 CPU 的 kill 语义优先。

---

## 七、异常与精确性建议

### 1. 第一版优先追求“可控精确”

建议第一版先采用保守策略：

- 协处理器异常必须在 CPU 可识别的提交边界生效；
- 出现异常后，由 CPU 统一执行 flush / trap；
- 不要求第一版就支持复杂的部分提交模型。

### 2. 异常最少应区分

- 非法向量指令；
- 协处理器 busy/不可用；
- 向量访存异常；
- 内部错误或未实现功能；
- 可恢复与不可恢复异常。

### 3. 调试信息建议保留

建议异常上报时至少保留：

- 对应 `pc`；
- `seq`；
- `cause`；
- 附加 `tval`。

这样在 commit trace 中更容易对齐分析。

---

## 八、访存接口建议

向量访存是后续最容易复杂化的部分，因此建议分阶段处理。

### 第一版建议

第一版不要急着让向量协处理器直接深入改造现有 LSU，而是先在接口层约定两种可能模式：

1. **保守模式**：协处理器暂不支持独立访存，只做纯计算；
2. **过渡模式**：协处理器通过一个独立 memory request/response 口向 CPU/LSU 请求服务。

### 第一版不建议

- 直接让协处理器绕开 CPU 自己改 AXI 主设备；
- 直接把向量访存硬塞进当前 LSU 特判分支；
- 在没有统一 kill/exception 语义前做复杂多请求并发。

### 后续方向

如果后面要支持真正的向量 load/store，建议最终走向：

- `cop_mem_req_valid/ready/payload`
- `cop_mem_resp_valid/ready/payload`

并让 CPU 侧或共享 LSU 统一仲裁 DCache/Xbar 访问。

---

## 九、建议的第一版实现范围

为了让 CPU 团队和向量团队尽快分工，第一版建议只覆盖以下最小闭环。

### CPU 侧第一版

- 预留协处理器 issue/resp/kill/busy/exception 接口；
- 增加一个最小在飞项记录；
- 在 flush 路径上把 kill 语义接通；
- 在 WBU/commit 路径上允许协处理器结果写回；
- 对未实现向量指令先做 trap 或保守拦截。

### 向量侧第一版

- 实现一个 `dummy coprocessor`；
- 支持接收请求、延迟若干拍、返回一个确定结果；
- 支持在 kill 后丢弃结果；
- 支持至少一种简单异常上报。

### 第一版验证目标

- 标量已有测试不回退；
- 协处理器请求可被成功发射和返回；
- kill 后不会错误写回；
- 异常能进入统一 trap/flush 流程。

---

## 十、后续细化时要补的内容

后续精细化时建议补全：

1. tag/seq 的精确定义；
2. 是否允许多个未完成请求在飞；
3. 向量寄存器文件接口；
4. 向量 CSR 模型；
5. 向量访存请求格式；
6. 与 scoreboard 的配合方式；
7. commit trace 中向量事件的记录格式。

## 十一、第一版 RTL 冻结建议

前文故意保留了一些扩展空间，但如果现在要尽快开始 CPU 侧和向量侧并行开发，建议把**第一版真正冻结**为下面这组更小、更明确的接口。

这一版的目标不是“一步到位支持完整向量机”，而是：

- 足够小，CPU 很快能开口子；
- 足够稳，向量侧可以独立实现 `dummy coprocessor`；
- 足够清晰，后续扩展不会推翻控制语义。

### 1. 第一版范围收敛

第一版建议明确限制为：

1. **单发射、单未完成协处理器请求**；
2. **不支持向量独立访存**；
3. **不单独开异常通道**，异常并入最终 `response`；
4. **不允许协处理器直接修改 GPR/CSR/内存**；
5. **所有架构可见提交都由 CPU 完成**；
6. **不支持协处理器主动 redirect PC**；
7. **kill 先支持粗粒度全杀**，不先做精细 tag kill。

这意味着第一版本质上是：

> CPU 向协处理器发一条长延迟请求，协处理器最终返回“完成 / 写回 / 异常”三者之一，而提交权仍在 CPU。

### 2. 第一版推荐端口

建议第一版只保留下面这些信号。

#### CPU -> Coprocessor

- `o_cop_issue_valid`
- `i_cop_issue_ready`
- `o_cop_issue_pc[31:0]`
- `o_cop_issue_insn[31:0]`
- `o_cop_issue_rs1_value[31:0]`
- `o_cop_issue_rs2_value[31:0]`
- `o_cop_issue_rd[4:0]`
- `o_cop_issue_rd_we`
- `o_cop_issue_tag[3:0]`
- `o_cop_kill`

#### Coprocessor -> CPU

- `i_cop_resp_valid`
- `o_cop_resp_ready`
- `i_cop_resp_tag[3:0]`
- `i_cop_resp_rd[4:0]`
- `i_cop_resp_rd_we`
- `i_cop_resp_wdata[31:0]`
- `i_cop_resp_status[1:0]`
- `i_cop_resp_cause[5:0]`
- `i_cop_resp_tval[31:0]`
- `i_cop_busy`

### 3. 为什么删掉了一些前文信号

为尽快落 RTL，第一版建议先**不要**单独实现：

- `cop_idle`：可以由 `!cop_busy` 推导；
- `cop_fault_pending`：第一版直接并入 `resp_status`；
- `cop_exc_*`：第一版不单独开异常上报通道；
- `cop_kill_seq` / `cop_kill_all`：第一版先统一成一个 `o_cop_kill`；
- `cop_mem_req/resp_*`：访存接口放到第二阶段之后。

这样 CPU 侧接口数量明显更小，也更利于先打通最小闭环。

### 4. 第一版字段语义

#### `o_cop_issue_pc`

- 原始指令 PC；
- 用于调试、异常定位、trace 对齐。

#### `o_cop_issue_insn`

- 原始 32-bit 指令编码；
- 推荐让协处理器自己完成进一步子解码，而不是 CPU 提前把所有向量语义拆碎。

#### `o_cop_issue_rs1_value` / `o_cop_issue_rs2_value`

- 由 CPU 提供的标量源操作数值；
- 第一版先不额外传 `rs1/rs2` 编号；
- 如果后续确实需要 debug 再补编号字段。

#### `o_cop_issue_rd` / `o_cop_issue_rd_we`

- 表示该指令在架构上是否需要一个标量写回目标；
- 若某些向量指令只更新协处理器内部状态，则 `rd_we = 0`。

#### `o_cop_issue_tag`

- 预留给后续扩展；
- 第一版虽然只允许一个请求在飞，但仍建议保留 4-bit tag；
- CPU 侧可以简单递增，协处理器原样带回。

#### `i_cop_resp_status`

建议第一版编码为：

- `2'b00`：正常完成；
- `2'b01`：异常完成；
- `2'b10`：保留；
- `2'b11`：保留。

第一版不建议引入更复杂的“部分完成”或“重试”语义。

#### `i_cop_resp_cause` / `i_cop_resp_tval`

- 仅在 `resp_status == 2'b01` 时有效；
- 由 CPU 在 commit/trap 路径中使用。

### 5. 第一版严格行为约束

建议双方在第一版实现中共同遵守下面几条硬约束。

#### 约束 A：同一时刻最多一个协处理器请求在飞

- 当上一条协处理器请求尚未完成或尚未被 kill 清理前；
- CPU 不再发第二条协处理器请求。

这是当前最值得做的收敛点，因为它能大幅降低：

- commit 复杂度；
- kill 边界复杂度；
- 异常精确性复杂度。

#### 约束 B：每个已接收请求只能有一个终止事件

一个被 `issue_valid && issue_ready` 接收的请求，最终只能以下二选一结束：

1. 收到一次有效 `response`；
2. 被一次 `kill` 取消。

不允许：

- kill 后仍返回有效写回；
- 返回两次；
- 既被 kill 又晚到一拍再次提交。

#### 约束 C：kill 优先级高于 response

如果 `kill` 与某条被 kill 的旧请求返回发生竞争：

- CPU 以 kill 语义优先；
- 协处理器必须保证该请求不会形成架构可见提交；
- 必要时可允许 response 被 CPU 直接丢弃。

#### 约束 D：协处理器只“完成”，不“提交”

- 协处理器的 `response` 仅代表结果或异常已经准备好；
- 最终写回、trap、生效时机由 CPU 决定；
- 协处理器不直接改写 GPR、CSR 或 PC。

#### 约束 E：第一版不做访存

- 协处理器第一版只做纯计算或状态类操作；
- 所有真实向量 load/store 留到第二阶段之后。

### 6. CPU 侧最小实现建议

CPU 第一版建议最少补下面这些逻辑。

1. 在 decode/issue 路径识别 `is_cop_insn`；
2. 当协处理器忙时，阻止新的协处理器指令发射；
3. 记录一个最小在飞项：`valid/tag/rd/rd_we/pc`；
4. 在全局 flush/trap/mispredict 时产生 `o_cop_kill`；
5. 在收到 `resp_status == OK` 时，走正常写回/完成路径；
6. 在收到 `resp_status == EXC` 时，走统一 trap 路径。

第一版甚至可以允许 CPU 直接把“协处理器请求在飞”当作一个特殊的多周期执行项处理，而不急着完整引入 scoreboard。

### 7. 向量侧最小实现建议

向量侧第一版建议最少实现：

1. 接收 `issue` 后锁存输入；
2. 延迟固定若干拍后给出 `response`；
3. 若期间收到 `kill`，丢弃这条请求；
4. `busy` 在接收请求后拉高，在 `response` 或 `kill` 清理后拉低；
5. 至少支持一种正常完成和一种异常完成。

这一版不要求：

- 完整向量寄存器文件；
- 完整向量 ISA；
- 多 lane；
- 高吞吐。

### 8. 第一版最小联调用例

建议第一批就写出下面 5 类联调场景：

1. 一条协处理器指令正常返回并写回 GPR；
2. 一条协处理器指令正常返回但 `rd_we = 0`；
3. 一条协处理器指令执行中被 `kill`；
4. 一条协处理器指令返回异常；
5. 连续两条协处理器指令串行执行。

如果这 5 类都能稳定通过，再进入下一轮接口细化会更稳。

## 十二、结论

当前最合适的策略不是立即把向量处理器细节做满，而是先约定一个足够稳定的后端接口框架：

1. CPU 负责统一 issue / commit / flush / trap；
2. 协处理器作为长延迟执行后端接入；
3. 第一版先用 valid/ready + kill + exception 的最小集合建立闭环；
4. 之后再逐步细化访存、CSR、并发在飞和性能问题。

这样就可以比较自然地实现分工：

- 一个人推进 CPU 结构升级；
- 一个人推进向量协处理器本体；
- 双方围绕同一份接口文档并行工作。
