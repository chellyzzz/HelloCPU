# HelloCPU 向量协处理器接口草案

## 一、文档目标

本文档给出 HelloCPU 与未来向量协处理器之间的一版**粗接口草案**。

目标不是一次性把所有细节定死，而是先统一以下内容：

1. CPU 与向量协处理器之间的边界；
2. issue / complete / commit / kill 的基本语义；
3. 允许 CPU 侧和向量侧并行开发的最小接口集合；
4. 后续细化时尽量不推翻当前框架。

本文档强调的是 **接口语义先行，具体实现后定**。

---

## 二、设计原则

### 1. 协处理器是“执行后端”，不是特殊函数调用

向量协处理器在系统中的定位应当更接近：

- 一个新的 issue target；
- 一个新的长延迟结果来源；
- 一个新的异常来源；
- 未来可能具备独立访存能力的后端单元。

不建议把它设计成 EXU 里的一堆特殊 if/else 路径，否则后续在：

- flush；
- trap；
- 异常恢复；
- 访存仲裁；

这些问题上会迅速失控。

### 2. 先冻结接口语义，不冻结内部实现

当前阶段先定义：

- 请求长什么样；
- 返回长什么样；
- 什么情况下 kill；
- 什么情况下允许提交；
- 异常如何上报。

而不急着定义：

- 向量寄存器文件具体结构；
- lane 数量；
- 向量长度编码；
- 精确到 RTL 级的内部流水实现。

### 3. 控制流恢复语义必须统一

无论来自：

- 标量分支误预测；
- ecall / mret / trap；
- 协处理器异常；
- 向量访存异常；

都应尽量走统一的 kill / flush / redirect / commit 语义。

### 4. 第一版接口优先支持单发射顺序核

HelloCPU 当前是单发射顺序核，因此第一版接口默认：

- CPU 每拍最多发出一条协处理器请求；
- 协处理器可以多拍完成；
- CPU 保留顺序提交语义；
- 不要求第一版就支持乱序返回后直接乱序提交。

但接口字段中应预留 tag / sequence 信息，避免以后重构。

---

## 三、系统角色划分

### CPU 负责

- 指令取值、解码、标量寄存器读写；
- 指令是否发往协处理器的判定；
- 发射时携带源操作数和控制信息；
- 统一管理 flush / redirect / kill；
- 统一决定什么时候可以架构提交；
- 接收协处理器结果并写回；
- 接收协处理器异常并转入 trap 处理。

### 协处理器负责

- 接收 issue 请求；
- 在内部完成向量运算；
- 在需要时发起或请求访存；
- 返回结果或完成状态；
- 报告异常；
- 在收到 kill 后丢弃被取消的未完成操作。

### 暂不在第一版定死的边界

- 向量访存是否独立穿过 LSU；
- 向量 CSR 是否完全采用标准 RISC-V V 规范；
- 是否支持多个未完成向量操作同时在飞。

---

## 四、建议的最小接口集合

第一版建议至少定义以下接口。

### 1. Issue 请求通道

采用 valid/ready 握手：

- `cop_issue_valid`
- `cop_issue_ready`
- `cop_issue_payload`

语义：

- CPU 在解码/发射阶段确认该指令由协处理器处理时，拉高 `cop_issue_valid`；
- 协处理器可接收请求时拉高 `cop_issue_ready`；
- `valid && ready` 的那个拍，表示该请求正式进入协处理器。

### 2. Response 返回通道

采用 valid/ready 握手：

- `cop_resp_valid`
- `cop_resp_ready`
- `cop_resp_payload`

语义：

- 协处理器在某条请求完成、可回传结果或完成状态时拉高 `cop_resp_valid`；
- CPU 在可以接收返回结果时拉高 `cop_resp_ready`；
- `valid && ready` 的那个拍，表示返回被 CPU 接收。

### 3. Kill / Flush 控制通道

- `cop_kill_valid`
- `cop_kill_seq`
- `cop_kill_all`

语义：

- 当 CPU 因误预测、异常、trap、mret 等事件取消未提交指令时，向协处理器发送 kill；
- `cop_kill_all` 表示当前所有未提交向量操作全部失效；
- `cop_kill_seq` 表示只 kill 某个序号之后或某个标签对应的操作，具体编码可在细化版确定。

第一版实现可以先只支持粗粒度的 `kill_all`。

### 4. Busy / Status 状态接口

- `cop_busy`
- `cop_idle`
- `cop_fault_pending`

语义：

- `cop_busy` 表示协处理器内部仍有未完成工作；
- `cop_idle` 可视为 `!cop_busy`，是否单独保留实现可后定；
- `cop_fault_pending` 表示已有异常待 CPU 接收。

### 5. Exception 上报接口

- `cop_exc_valid`
- `cop_exc_cause`
- `cop_exc_tval`
- `cop_exc_seq`

语义：

- 协处理器在发现需要架构可见的异常时，通过该通道报告；
- CPU 负责把异常纳入统一 trap 流程；
- `seq` 用于标识异常对应的在飞操作。

---

## 五、建议的 payload 字段

### 1. `cop_issue_payload`

第一版建议至少包含：

- `opcode`：向量操作类型或子命令；
- `funct`：更细粒度的子操作编码；
- `src0_value`：标量源操作数 0；
- `src1_value`：标量源操作数 1；
- `src2_value`：可选第三源操作数；
- `dst_type`：结果写回目标类型；
- `dst_idx`：目标寄存器编号或目标槽位；
- `seq`：该请求的顺序号或 tag；
- `pc`：原始指令 PC，便于调试和异常定位；
- `instr`：原始指令编码，便于早期 bring-up；
- `rm_or_mode`：模式位，保留给舍入、掩码、长度等扩展字段；
- `flags`：是否写回、是否需要提交确认、是否可能访存等控制位。

第一版不强制区分“向量寄存器源操作数”和“标量立即数”的最终编码方式，但要预留统一字段。

### 2. `cop_resp_payload`

第一版建议至少包含：

- `seq`：返回对应哪条请求；
- `dst_type`：结果写回目标类型；
- `dst_idx`：目标编号；
- `result0`：主结果数据；
- `result1`：可选附加结果；
- `wb_enable`：是否需要 CPU 执行写回；
- `done`：该请求是否彻底完成；
- `flags`：饱和、比较结果、部分完成等状态位；
- `exc`：是否同时携带异常。

第一版可以只真正使用 `result0`，其他字段先预留。

## 六、CPU 侧控制语义建议

### 1. Issue 语义

- CPU 只有在该指令尚未被 flush 且协处理器 ready 时，才允许发射；
- 发射成功后，该操作进入“在飞”状态；
- CPU 需要记录该操作是否尚未完成、是否尚未提交、是否已被 kill。

### 2. Complete 语义

- 协处理器返回 `resp` 只表示“执行结果已经准备好”；
- 不必等同于“已经架构提交”；
- CPU 可先缓存结果，再在满足顺序提交条件时正式写回。

这一区分非常重要，因为未来需要统一：

- 标量多周期执行；
- 向量长延迟执行；
- trap / mispredict 下的 kill；
- 精确异常。

### 3. Commit 语义

- 第一版仍建议由 CPU 统一提交；
- 协处理器不直接修改架构可见状态；
- 任何写回、CSR 影响、异常生效，都应由 CPU 在 commit 点统一决定。

### 4. Kill 语义

- kill 作用于“已发射但尚未提交”的向量操作；
- 被 kill 的操作即使内部算完，也不应再产生架构可见写回；
- 若返回与 kill 同时竞争，以 CPU 的 kill 语义优先。

---

## 七、异常与精确性建议

### 1. 第一版优先追求“可控精确”

建议第一版先采用保守策略：

- 协处理器异常必须在 CPU 可识别的提交边界生效；
- 出现异常后，由 CPU 统一执行 flush / trap；
- 不要求第一版就支持复杂的部分提交模型。

### 2. 异常最少应区分

- 非法向量指令；
- 协处理器 busy/不可用；
- 向量访存异常；
- 内部错误或未实现功能；
- 可恢复与不可恢复异常。

### 3. 调试信息建议保留

建议异常上报时至少保留：

- 对应 `pc`；
- `seq`；
- `cause`；
- 附加 `tval`。

这样在 commit trace 中更容易对齐分析。

---

## 八、访存接口建议

向量访存是后续最容易复杂化的部分，因此建议分阶段处理。

### 第一版建议

第一版不要急着让向量协处理器直接深入改造现有 LSU，而是先在接口层约定两种可能模式：

1. **保守模式**：协处理器暂不支持独立访存，只做纯计算；
2. **过渡模式**：协处理器通过一个独立 memory request/response 口向 CPU/LSU 请求服务。

### 第一版不建议

- 直接让协处理器绕开 CPU 自己改 AXI 主设备；
- 直接把向量访存硬塞进当前 LSU 特判分支；
- 在没有统一 kill/exception 语义前做复杂多请求并发。

### 后续方向

如果后面要支持真正的向量 load/store，建议最终走向：

- `cop_mem_req_valid/ready/payload`
- `cop_mem_resp_valid/ready/payload`

并让 CPU 侧或共享 LSU 统一仲裁 DCache/Xbar 访问。

---

## 九、建议的第一版实现范围

为了让 CPU 团队和向量团队尽快分工，第一版建议只覆盖以下最小闭环。

### CPU 侧第一版

- 预留协处理器 issue/resp/kill/busy/exception 接口；
- 增加一个最小在飞项记录；
- 在 flush 路径上把 kill 语义接通；
- 在 WBU/commit 路径上允许协处理器结果写回；
- 对未实现向量指令先做 trap 或保守拦截。

### 向量侧第一版

- 实现一个 `dummy coprocessor`；
- 支持接收请求、延迟若干拍、返回一个确定结果；
- 支持在 kill 后丢弃结果；
- 支持至少一种简单异常上报。

### 第一版验证目标

- 标量已有测试不回退；
- 协处理器请求可被成功发射和返回；
- kill 后不会错误写回；
- 异常能进入统一 trap/flush 流程。

---

## 十、后续细化时要补的内容

后续精细化时建议补全：

1. tag/seq 的精确定义；
2. 是否允许多个未完成请求在飞；
3. 向量寄存器文件接口；
4. 向量 CSR 模型；
5. 向量访存请求格式；
6. 与 scoreboard 的配合方式；
7. commit trace 中向量事件的记录格式。

---

## 十一、结论

当前最合适的策略不是立即把向量处理器细节做满，而是先约定一个足够稳定的后端接口框架：

1. CPU 负责统一 issue / commit / flush / trap；
2. 协处理器作为长延迟执行后端接入；
3. 第一版先用 valid/ready + kill + exception 的最小集合建立闭环；
4. 之后再逐步细化访存、CSR、并发在飞和性能问题。

这样就可以比较自然地实现分工：

- 一个人推进 CPU 结构升级；
- 一个人推进向量协处理器本体；
- 双方围绕同一份接口文档并行工作。
