# HelloCPU CPU 与向量协处理器联合发展规划

## 一、文档目标

本文档用于明确 HelloCPU 接下来一段时间内，CPU 侧与向量协处理器侧的联合推进路线。

核心目标不是单点做快，而是让两条开发线可以并行推进、边界稳定、联调风险可控。

---

## 二、当前判断

从当前项目状态看，HelloCPU 已经具备：

- 稳定可运行的五级顺序标量核；
- RV32IM + Zicsr；
- ICache / DCache；
- 基本可用的分支预测；
- CoreMark 正确通过。

因此项目已经不处于“先证明 CPU 能不能跑”的阶段，而进入：

- 标量 CPU 结构升级；
- 向量协处理器接口预留；
- 双线并行开发准备。

当前最大问题也不是 cache 容量或总线带宽，而是：

- 流水线停顿占比高；
- LSU、旁路、多周期单元和控制恢复边界仍较紧耦合；
- 未来如果直接接入向量后端，控制复杂度会上升很快。

---

## 三、总策略

建议接下来采用“三步走”。

### 第一步：先冻结粗接口

先用一份草案把 CPU 与向量协处理器之间的：

- issue；
- response；
- kill；
- exception；
- busy；
- memory 扩展方向；

这些边界约定下来。

这一步的目的不是把所有位宽和编码一次性定死，而是让双方不再各自假设。

### 第二步：CPU 与向量侧并行实现最小闭环

CPU 侧负责把结构接入口打通；
向量侧负责提供一个可联调的 dummy 协处理器。

这一步的目标是先打通：

- 发射；
- 完成；
- kill；
- 异常；
- 提交。

### 第三步：再各自深化

待最小闭环稳定后：

- CPU 侧继续推进 LSU、旁路、执行接口重构；
- 向量侧继续推进真实执行、状态、访存；
- 双方通过接口文档和 trace 机制联调。

---

## 四、CPU 侧主线任务

### 1. 重构 EXU 视角下的执行后端边界

目标：

- 把 ALU、LSU、MUL、DIV，以及未来的协处理器都视作“执行后端”；
- 统一 request / response / status 语义；
- 减少 EXU 内部的特殊分支控制。

### 2. 统一 commit / flush / kill / redirect 语义

目标：

- 标量分支误预测；
- trap / ecall / mret；
- 协处理器 kill；
- 向量异常；

尽量走一套稳定框架。

### 3. 明确在飞项管理

目标：

- 先支持最小的协处理器在飞项记录；
- 后续为 scoreboard 或多未完成请求预留空间。

### 4. 为后续性能优化打基础

目标：

- 更清晰地区分“结果已完成”和“结果已提交”；
- 为 LSU 重构、旁路增强、多周期单元局部化阻塞创造条件。

---

## 五、向量侧主线任务

### 1. 先做 dummy 协处理器

目标：

- 快速提供可联调对象；
- 验证握手、kill、异常、返回时序。

### 2. 再做最小真实向量能力

目标：

- 先支持少量纯计算型向量操作；
- 不在第一阶段引入复杂访存。

### 3. 最后扩到状态与访存

目标：

- 定义向量状态；
- 定义 CSR；
- 定义 memory request/response 扩展；
- 建立更真实的软件可见语义。

---

## 六、分工建议

### CPU 负责人

负责：

- 接口在 CPU 顶层和 EXU/WBU 路径中的接入；
- 在飞项和提交框架；
- kill/flush/exception 联动；
- 保证现有标量验证不回退。

### 向量负责人

负责：

- 协处理器模块本体；
- 内部状态机；
- 返回结果协议；
- kill 响应；
- 向量路径调试信号与后续功能扩展。

### 双方共同负责

- 接口文档维护；
- 指令编码与软件可见语义；
- trace 格式；
- 异常定义；
- 联调用例。

---

## 七、建议的阶段里程碑

### 里程碑 M1：接口冻结

完成标准：

- 接口草案形成文档；
- 双方确认第一版字段和语义；
- 明确第一版不做什么。

### 里程碑 M2：最小闭环跑通

完成标准：

- CPU 能发射协处理器请求；
- dummy 协处理器能返回；
- kill 和异常链路可验证；
- 标量已有测试无回退。

### 里程碑 M3：真实向量雏形

完成标准：

- 至少一类真实向量算子可运行；
- trace 能覆盖 issue/complete/commit；
- flush 场景无明显错误提交。

### 里程碑 M4：访存与状态扩展

完成标准：

- 向量状态定义基本稳定；
- 向量访存接口原型跑通；
- 异常路径与 CPU 统一。

### 里程碑 M5：性能优化阶段

完成标准：

- CPU 侧 LSU/旁路/阻塞优化开始受益于结构重构；
- 向量侧开始做 lane、并发、吞吐优化；
- 性能计数器和 trace 能支撑分析。

---

## 八、当前推荐的近期任务顺序

建议接下来优先做：

1. 确认 `docs/vector-coprocessor-interface.md`；
2. 先冻结 `V1 RTL` 端口和行为约束；
3. 向量侧实现 `dummy coprocessor`；
4. CPU 侧接入最小协处理器端口；
5. 增加最小联调用例；
6. 再进入 CPU 侧 LSU/执行接口重构；
7. 与此同时向量侧推进最小真实向量执行。

这个顺序的好处是：

- 可以尽快开始并行开发；
- 可以尽早暴露接口歧义；
- 不会把双方都卡在“大重构完成后再联调”。

---

## 九、结论

当前最合适的项目发展方向不是“先把 CPU 或向量某一边做得很深”，而是：

1. 先建立稳定接口；
2. 再打通最小闭环；
3. 然后 CPU 和向量两条线并行深化。

这样能让 HelloCPU 后续演进从“单核局部优化”转向“可扩展的标量核 + 协同向量后端”架构。
