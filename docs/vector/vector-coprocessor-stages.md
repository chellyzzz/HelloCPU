# HelloCPU 向量协处理器阶段建议

## 一、文档目标

本文档给出向量协处理器侧的阶段性开发建议，目标是帮助向量负责人在不阻塞 CPU 主线重构的前提下，分阶段把协处理器从"空壳接口"推进到**兼容 RISC-V Vector（RVV）标准的向量后端**。

### 最终目标

实现一个兼容 RVV 标准的向量协处理器，支持：
- 标准 RVV 指令编码（`vsetvl`/`vsetvli`、向量 load/store、向量计算）
- 32 个向量寄存器（v0-v31），可配置元素宽度（8/16/32/64-bit）
- 可配置向量长度（VL），通过 `vsetvl`/`vsetvli` 动态设置
- 向量掩码（vm=0/1，v0 作掩码）
- 向量访存（stride/gather/scatter）

### 当前阶段

当前实现是基于 `custom-0`（`opcode=0x0b`）的自定义协处理器原型，用于验证 COP 接口、状态管理和 VRF 基础设施。**不是 RVV 兼容实现**，但为后续 RVV 迁移打下基础。

---

## 二、总体策略

建议向量侧采用：

1. **先打通接口闭环**；
2. **再做控制正确性**；
3. **最后做访存和性能**。

不要一开始就把重点放在：

- 完整向量 ISA；
- 复杂 lane 结构；
- 多请求并发；
- 高吞吐访存。

因为当前项目的主风险还不是“算得不够快”，而是：

- CPU/协处理器边界没统一；
- kill/flush/exception 语义没走通；
- 调试基础设施还没覆盖向量路径。

---

## 三、阶段划分

### 阶段 0：接口对齐与占位实现

目标：

- 与 CPU 侧对齐第一版接口；
- 做一个最小可综合/可仿真的协处理器壳子；
- 先确保系统接得进去。

建议内容：

- 固定 `issue/resp/kill/busy/exception` 端口；
- 实现 `ready` 常高或有限 backpressure 的最小行为；
- 对收到的请求仅做简单寄存和回传；
- 提供可观测的 debug 状态。

验收标准：

- CPU 能成功向协处理器发请求；
- 协处理器能在若干拍后返回；
- 工程可以稳定编译、仿真、trace。

### 阶段 1：Dummy 功能闭环

目标：

- 建立真正可验证的功能闭环；
- 验证时序、kill、异常、写回路径。

建议内容：

- 支持 1 到 2 个简单伪操作；
- 例如 passthrough、标量加法扩展、固定模式向量填充值；
- 支持可配置延迟返回；
- 支持收到 kill 后丢弃未提交结果；
- 支持一类可控异常。

验收标准：

- 正常返回结果时可写回正确目的寄存器；
- kill 后不会发生错误写回；
- 异常可被 CPU 统一接住并 trap。

### 阶段 2：最小真实向量执行模型

目标：

- 从“演示协处理器”进入“真实向量后端雏形”；
- 先做纯计算，不碰复杂访存。

建议内容：

- 定义内部 micro-op 或简单执行控制；
- 支持少量纯计算型指令；
- 建立最小向量状态保存结构；
- 增加 busy 生命周期管理；
- 明确结果在 complete 与 commit 之间的边界。

建议首批功能：

- 向量加法；
- 向量按元素逻辑运算；
- 向量与标量混合操作中的一小部分；
- 简单 mask 或全使能模式。

验收标准：

- 能独立跑通几条最小向量程序；
- trace 中可看到发射、完成、提交闭环；
- flush 场景下无错误状态残留。

当前占位 COP 操作编码：`opcode=0x0b`，`funct3=0` 为标量运算（`funct7=0` 加法，`funct7=1` 减法，`funct7=2` 乘法低 32 位），`funct3=1` 为 4x8-bit lane add，`funct3=2` 为 4x8-bit lane xor，`funct3=3` 为 4x8-bit lane and，`funct3=4` 返回旧 scratch 并写入新 scratch，`funct3=5` 写入 vlen 并返回旧 vlen，`funct3=6` 只读 vlen，`funct3=7` 返回已完成 COP 操作计数。标量运算通过 funct7 扩展子操作；lane ops 和状态操作仍使用 funct3 编码。

### 阶段 3：向量状态与 CSR 细化

目标：

- 让协处理器从“能算”进入“有架构状态”；
- 形成可持续扩展的软件可见语义。

建议内容：

- 定义最小 CSR 或配置寄存器；
- 明确向量长度、模式、状态位；
- 明确 busy、fault、done 对软件是否可见；
- 明确 reset、trap、mret 后状态恢复规则。

验收标准：

- 状态切换语义稳定；
- CPU 与协处理器对 CSR/状态机理解一致；
- 调试时可以定位向量状态变化。

### 阶段 4：引入向量访存

目标：

- 打通真实系统里最难的部分之一；
- 让协处理器从纯计算扩展到可用的数据处理单元。

建议内容：

- 引入独立 `mem_req/mem_resp` 抽象；
- 先做串行、低并发、可精确 kill 的版本；
- 明确未完成访存与 flush 的交互；
- 明确访存异常和 `tval` 回报机制。

验收标准：

- 向量 load/store 至少能在低并发场景下正确运行；
- miss、flush、异常时不会污染架构状态；
- 能和 CPU 标量访存共享仲裁而不破坏原有功能。

### 阶段 5：性能优化与结构扩展

目标：

- 在正确性与接口都稳定后，再做吞吐优化。

建议内容：

- lane 化；
- 更深流水；
- 多请求在飞；
- 访存合并；
- 更细粒度 scoreboard；
- 更完善的向量性能计数器。

验收标准：

- 吞吐提升有性能计数器支撑；
- 不破坏 kill/flush/exception 统一语义；
- 不回退已有标量和向量验证结果。

---

## 四、向量负责人当前最应该先做什么

如果现在马上开始分工，向量侧最值得优先推进的是：

1. 基于接口草案实现一个 `dummy coprocessor`；
2. 明确内部状态机如何响应 `issue`、`kill`、`resp`；
3. 约定最小 trace 观测点；
4. 设计第一批最简单的“演示型向量操作”。

不要把第一周时间花在：

- 完整向量 ISA 解码；
- 复杂寄存器文件；
- 高性能访存；
- 多 lane 并发优化。

---

## 五、建议的验证顺序

建议向量侧按下面顺序自测：

1. 请求发出后固定延迟返回；
2. 返回结果写回标量目标；
3. kill 打断未完成返回；
4. 异常上报与 CPU trap 联动；
5. 连续两条协处理器请求；
6. 协处理器忙时的 backpressure；
7. flush 与返回同拍竞争；
8. 后续再加入访存。

---

## 六、结论

向量侧最合理的推进方式不是“先做大而全的向量机”，而是：

1. 先成为一个行为清晰的长延迟执行后端；
2. 先证明自己能正确接入 CPU 的 issue/kill/commit 体系；
3. 再逐步长出状态、访存和性能能力。

这样既能和 CPU 升级并行推进，也能显著降低联调风险。

## 七、当前实现与 RVV 标准的差距

### 7.1 当前实现（自定义 COP）

| 维度 | 当前状态 |
|------|----------|
| 指令编码 | `custom-0`（`opcode=0x0b`），自定义 funct3/funct7 |
| 寄存器文件 | 4 个 32-bit 寄存器（v0-v3），固定 4x8-bit lane |
| 元素宽度 | 固定 8-bit |
| 向量长度 | 固定 4 元素 |
| 掩码 | 无 |
| 访存 | 无向量访存 |
| 配置 | 简单 `vlen` 寄存器 |

### 7.2 RVV 标准要求

| 维度 | RVV 标准 |
|------|----------|
| 指令编码 | 标准 RVV opcodes（OP-V、LOAD-FP 等） |
| 寄存器文件 | 32 个向量寄存器（v0-v31），宽度可配置 |
| 元素宽度 | 8/16/32/64-bit 可配置 |
| 向量长度 | 通过 `vsetvl`/`vsetvli` 动态配置 |
| 掩码 | 支持向量掩码（vm=0/1，v0 作掩码） |
| 访存 | 向量 load/store（stride/gather/scatter） |
| 配置 | `vsetvl`/`vsetvli` 指令 |

### 7.3 差距总结

1. **编码差距**：需要从 `custom-0` 迁移到标准 RVV opcodes
2. **寄存器差距**：需要从 4 个扩展到 32 个，支持可配置元素宽度
3. **配置差距**：需要实现 `vsetvl`/`vsetvli` 指令
4. **掩码差距**：需要实现向量掩码机制
5. **访存差距**：需要实现向量 load/store 指令

## 八、RVV 迁移技术路线

### 阶段 A：当前自定义 COP（已完成）

- 验证 COP 接口（issue/resp/kill/flush）
- 验证状态管理（scratch/vlen/opcount）
- 验证 VRF 基础设施（vload/vstore/lane ops）
- 验证 flush 语义和时序

### 阶段 B：RVV 编码迁移（下一步）

目标：将自定义编码迁移到标准 RVV 编码，同时保持 CPU 侧接口不变。

建议内容：
1. **实现 `vsetvl`/`vsetvli` 指令**：
   - 在 CPU 侧解码 `vsetvl`/`vsetvli` 指令
   - 通过 COP 接口传递配置信息
   - COP 侧实现 `vtype` 和 `vl` 寄存器

2. **扩展 VRF 到 32 个寄存器**：
   - 将 VRF 从 4 个扩展到 32 个
   - 支持可配置元素宽度（8/16/32/64-bit）
   - 实现标准 RVV 寄存器编码（v0-v31）

3. **实现标准 RVV 算术指令**：
   - `vadd.vv`、`vsub.vv`、`vand.vv`、`vxor.vv` 等
   - 支持可配置元素宽度
   - 支持向量长度（VL）

4. **实现标准 RVV load/store 指令**：
   - `vle8.v`、`vle16.v`、`vle32.v`、`vle64.v`
   - `vse8.v`、`vse16.v`、`vse32.v`、`vse64.v`
   - 先实现 unit-stride，后续扩展 stride/gather/scatter

验收标准：
- 能运行标准 RVV 程序（如 RVV 向量加法示例）
- `vsetvl`/`vsetvli` 正确配置向量长度和元素宽度
- 向量 load/store 正确访问内存
- 向量算术指令正确计算结果

### 阶段 C：RVV 掩码和高级特性

目标：实现 RVV 掩码机制和高级特性。

建议内容：
1. **实现向量掩码**：
   - v0 作为掩码寄存器
   - 支持 `vm=0`（掩码模式）和 `vm=1`（无掩码模式）
   - 实现掩码生成指令（`vmseq.vv`、`vmslt.vv` 等）

2. **实现更多 RVV 指令**：
   - 向量乘法（`vmul.vv`）
   - 向量除法（`vdiv.vv`）
   - 向量归约（`vredsum.vs`）
   - 向量置换（`vmv.vv`）

3. **实现向量异常处理**：
   - 向量访存异常
   - 向量配置异常
   - 与 CPU 异常处理联动

验收标准：
- 支持 RVV 掩码操作
- 支持更多 RVV 指令
- 异常处理正确

### 阶段 D：性能优化

目标：优化 RVV 实现的性能。

建议内容：
1. **Lane 化**：实现多 lane 并行执行
2. **更深流水**：增加流水线深度，提高吞吐
3. **多请求在飞**：支持多个向量操作并发
4. **访存合并**：优化向量访存效率
5. **Scoreboard**：实现更细粒度的依赖跟踪

验收标准：
- 吞吐提升有性能计数器支撑
- 不破坏 RVV 语义
- 不回退已有测试结果

## 九、当前自定义 COP 到 RVV 的映射关系

| 当前操作 | RVV 对应 | 迁移策略 |
|----------|----------|----------|
| `funct3=0, funct7=0` (scalar add) | 标量 ALU | 保留，RVV 不需要 |
| `funct3=0, funct7=1` (scalar sub) | 标量 ALU | 保留，RVV 不需要 |
| `funct3=0, funct7=2` (scalar mul) | 标量 ALU | 保留，RVV 不需要 |
| `funct3=0, funct7=3` (vload) | `vle8.v` 等 | 迁移到标准 RVV load |
| `funct3=0, funct7=4` (vstore) | `vse8.v` 等 | 迁移到标准 RVV store |
| `funct3=0, funct7=5-11` (vrf lane ops) | `vadd.vv` 等 | 迁移到标准 RVV 算术 |
| `funct3=4` (scratch) | 无直接对应 | 保留为内部状态 |
| `funct3=5-6` (vlen) | `vsetvl`/`vsetvli` | 迁移到标准 RVV 配置 |
| `funct3=7` (opcount) | 无直接对应 | 保留为调试工具 |

---

## 十、总结

当前自定义 COP 是 RVV 迁移的**验证平台**，不是最终实现。通过当前阶段，我们验证了：
1. COP 接口的正确性
2. 状态管理的可行性
3. VRF 基础设施的可靠性
4. flush 语义的一致性

下一步是基于当前验证结果，开始 RVV 编码迁移，最终实现兼容 RVV 标准的向量协处理器。
