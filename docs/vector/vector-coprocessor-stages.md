# HelloCPU 向量协处理器阶段建议

## 一、文档目标

本文档给出向量协处理器侧的阶段性开发建议，目标是帮助向量负责人在不阻塞 CPU 主线重构的前提下，分阶段把协处理器从“空壳接口”推进到“可运行的真实后端”。

---

## 二、总体策略

建议向量侧采用：

1. **先打通接口闭环**；
2. **再做控制正确性**；
3. **最后做访存和性能**。

不要一开始就把重点放在：

- 完整向量 ISA；
- 复杂 lane 结构；
- 多请求并发；
- 高吞吐访存。

因为当前项目的主风险还不是“算得不够快”，而是：

- CPU/协处理器边界没统一；
- kill/flush/exception 语义没走通；
- 调试基础设施还没覆盖向量路径。

---

## 三、阶段划分

### 阶段 0：接口对齐与占位实现

目标：

- 与 CPU 侧对齐第一版接口；
- 做一个最小可综合/可仿真的协处理器壳子；
- 先确保系统接得进去。

建议内容：

- 固定 `issue/resp/kill/busy/exception` 端口；
- 实现 `ready` 常高或有限 backpressure 的最小行为；
- 对收到的请求仅做简单寄存和回传；
- 提供可观测的 debug 状态。

验收标准：

- CPU 能成功向协处理器发请求；
- 协处理器能在若干拍后返回；
- 工程可以稳定编译、仿真、trace。

### 阶段 1：Dummy 功能闭环

目标：

- 建立真正可验证的功能闭环；
- 验证时序、kill、异常、写回路径。

建议内容：

- 支持 1 到 2 个简单伪操作；
- 例如 passthrough、标量加法扩展、固定模式向量填充值；
- 支持可配置延迟返回；
- 支持收到 kill 后丢弃未提交结果；
- 支持一类可控异常。

验收标准：

- 正常返回结果时可写回正确目的寄存器；
- kill 后不会发生错误写回；
- 异常可被 CPU 统一接住并 trap。

### 阶段 2：最小真实向量执行模型

目标：

- 从“演示协处理器”进入“真实向量后端雏形”；
- 先做纯计算，不碰复杂访存。

建议内容：

- 定义内部 micro-op 或简单执行控制；
- 支持少量纯计算型指令；
- 建立最小向量状态保存结构；
- 增加 busy 生命周期管理；
- 明确结果在 complete 与 commit 之间的边界。

建议首批功能：

- 向量加法；
- 向量按元素逻辑运算；
- 向量与标量混合操作中的一小部分；
- 简单 mask 或全使能模式。

验收标准：

- 能独立跑通几条最小向量程序；
- trace 中可看到发射、完成、提交闭环；
- flush 场景下无错误状态残留。

当前占位 COP 操作编码：`opcode=0x0b`，`funct3=0` 为标量加法回环，`funct3=1` 为 4x8-bit lane add，`funct3=2` 为 4x8-bit lane xor，`funct3=3` 为 4x8-bit lane and。四者都仅使用 GPR 源/目的寄存器，不引入独立向量寄存器或访存状态。

### 阶段 3：向量状态与 CSR 细化

目标：

- 让协处理器从“能算”进入“有架构状态”；
- 形成可持续扩展的软件可见语义。

建议内容：

- 定义最小 CSR 或配置寄存器；
- 明确向量长度、模式、状态位；
- 明确 busy、fault、done 对软件是否可见；
- 明确 reset、trap、mret 后状态恢复规则。

验收标准：

- 状态切换语义稳定；
- CPU 与协处理器对 CSR/状态机理解一致；
- 调试时可以定位向量状态变化。

### 阶段 4：引入向量访存

目标：

- 打通真实系统里最难的部分之一；
- 让协处理器从纯计算扩展到可用的数据处理单元。

建议内容：

- 引入独立 `mem_req/mem_resp` 抽象；
- 先做串行、低并发、可精确 kill 的版本；
- 明确未完成访存与 flush 的交互；
- 明确访存异常和 `tval` 回报机制。

验收标准：

- 向量 load/store 至少能在低并发场景下正确运行；
- miss、flush、异常时不会污染架构状态；
- 能和 CPU 标量访存共享仲裁而不破坏原有功能。

### 阶段 5：性能优化与结构扩展

目标：

- 在正确性与接口都稳定后，再做吞吐优化。

建议内容：

- lane 化；
- 更深流水；
- 多请求在飞；
- 访存合并；
- 更细粒度 scoreboard；
- 更完善的向量性能计数器。

验收标准：

- 吞吐提升有性能计数器支撑；
- 不破坏 kill/flush/exception 统一语义；
- 不回退已有标量和向量验证结果。

---

## 四、向量负责人当前最应该先做什么

如果现在马上开始分工，向量侧最值得优先推进的是：

1. 基于接口草案实现一个 `dummy coprocessor`；
2. 明确内部状态机如何响应 `issue`、`kill`、`resp`；
3. 约定最小 trace 观测点；
4. 设计第一批最简单的“演示型向量操作”。

不要把第一周时间花在：

- 完整向量 ISA 解码；
- 复杂寄存器文件；
- 高性能访存；
- 多 lane 并发优化。

---

## 五、建议的验证顺序

建议向量侧按下面顺序自测：

1. 请求发出后固定延迟返回；
2. 返回结果写回标量目标；
3. kill 打断未完成返回；
4. 异常上报与 CPU trap 联动；
5. 连续两条协处理器请求；
6. 协处理器忙时的 backpressure；
7. flush 与返回同拍竞争；
8. 后续再加入访存。

---

## 六、结论

向量侧最合理的推进方式不是“先做大而全的向量机”，而是：

1. 先成为一个行为清晰的长延迟执行后端；
2. 先证明自己能正确接入 CPU 的 issue/kill/commit 体系；
3. 再逐步长出状态、访存和性能能力。

这样既能和 CPU 升级并行推进，也能显著降低联调风险。
