# HelloCPU 分支预测 Bug 定位与修复过程

## 一、文档目的

本文档记录 HelloCPU 在分支预测相关问题上的完整调试思路，重点不是简单罗列结论，而是整理：

1. 问题是如何一步一步被缩小的；
2. 为什么最后能定位到真正的根因；
3. 每一个关键修复点对应的具体现象和 RTL 问题是什么；
4. 这次调试对后续 CPU 演进有什么启发。

> 本文档是分支预测 bring-up 的历史调试记录，不维护当前 CoreMark 性能数字。最新性能结果请看 `coremark-results.md`，当前 CPU 优化优先级请看 `cpu-design-plan.md`。

---

## 二、最初现象

最开始开启完整分支预测后，CPU 并不是“结果算错但还能结束”，而是更糟糕：

- `make run` 可以通过；
- 但 CoreMark 在某些预测配置下会直接 **超时**；
- 在另一些裁剪后的预测配置下，CoreMark 可以跑完，但 **CRC 错误**。

这说明问题不只是单纯“预测率不够高”，而是：

> 分支预测路径里存在会污染架构状态或破坏控制流恢复的 bug。

换句话说，这不是“性能优化没做好”，而是“正确性 bug”。

---

## 三、第一步：先建立正确的对照基线

遇到这种问题，最忌讳直接在 RTL 里猜。

正确做法是先回答一个更基础的问题：

> 在完全不依赖预测器的情况下，CPU 本身是不是正确的？

因此第一步做的是：

- 关闭 IFU 预测路径；
- 重新运行 CoreMark；
- 检查 CRC 是否正确。

结果是：

- 无预测配置下 CoreMark CRC 正确；
- 程序可以正常结束；
- 这说明 LSU、ALU、MUL、DIV、cache、基本控制流总体上是可用的。

这个结论非常关键，因为它把问题范围从“整个 CPU”缩小成：

- **预测引入的控制流恢复问题**；
- 或者 **预测路径导致的错误提交问题**。

---

## 四、第二步：把“全预测问题”拆成多个小问题

完整预测路径同时包含：

- BTB 条件分支预测；
- static JAL 预测；
- RAS 返回预测。

如果在“全开”状态下直接调试，问题很容易混在一起。

所以调试时采取的是 **逐项裁剪**：

1. 全关预测；
2. 只开 BTB；
3. 只恢复部分预测；
4. 最后再恢复完整预测。

这一步得到一个非常有价值的中间状态：

### BTB-only 状态

- 条件分支预测保留；
- JAL 静态预测关闭；
- RAS 返回预测关闭；
- CoreMark **不再超时**；
- 但 CoreMark **CRC 错误**。

这说明问题成功从：

- “死循环 / 卡住”

缩小成：

- “错误路径提交导致最终结果错误”。

这比超时状态更容易定位，因为程序还能结束，能够拿到完整 trace。

---

## 五、第三步：不是只看最终结果，而是做提交级 trace 对比

接下来最关键的决定是：

> 不再靠肉眼看最后的 CRC 错误猜问题，而是直接比较有预测和无预测的**提交序列**。

因此在仿真层加入了辅助 trace：

- `--commit-trace`
- `--mem-trace`

记录信息包括：

- 提交 PC
- 写回寄存器号 `rd`
- 写回数据 `wdata`
- store 地址 / 数据 / byte mask
- 控制流类型
- 预测相关标志
- `pc_update`
- `flush`

调试的基本思路变成：

1. 生成 **无预测正确版 trace**；
2. 生成 **BTB-only 错误版 trace**；
3. 逐条比对；
4. 找到 **第一处真实架构分歧**。

这里“第一处真实架构分歧”非常重要。

因为很多表面差异其实不是 bug，例如：

- `mcycle` 不同；
- `start_time` / `stop_time` 不同；
- 输出顺序里与时间相关的值不同；

这些都必须先排除，不能把它们误当成根因。

---

## 六、第四步：找到 predicted-taken / actual-not-taken 漏恢复

在 BTB-only trace 里，最后找到了一类非常关键的现象：

- 某条条件分支 `pred=1`
- 但 `pcorr=0`
- 按理说这是误预测
- 可后续并没有看到应有的 `pcupdate/flush`

这就说明：

> CPU 明明知道这次预测错了，但恢复逻辑没有把前端拉回正确路径。

继续对照 RTL 后，发现 WBU 的恢复条件依赖的是：

- **实际 taken 的分支结果**

而不是：

- **这是一个分支指令并且预测错误**。

这会导致一种特殊但非常致命的情况：

### 场景

- IFU 预测条件分支 **taken**；
- EXU 实际计算结果是 **not taken**；
- 于是 actual taken = 0；
- WBU 看到 `i_brch = 0`，误以为“这不是需要恢复的控制流”；
- 最终不触发 redirect；
- CPU 沿着错误的 taken 目标继续执行。

### 根因本质

恢复条件写错了。

它不该问：

- “这条分支实际是否跳转？”

而应该问：

- “这是否是一条分支指令，并且预测是否正确？”

### 修复

修复分为两部分：

1. WBU 使用 `i_is_brch && !i_predict_correct` 判断分支是否需要恢复；
2. 分支提交路径传入 EXU 真实算出的 `redirect_pc`，这样 not-taken 时可以回到 `pc+4`。

这一步修完后，BTB-only 状态下 CoreMark 就从 CRC 错修复为 CRC 正确。

---

## 七、第五步：恢复 JAL / RAS 后又暴露出新的问题

BTB-only 修好之后，并不意味着完整预测已经稳定。

接下来把：

- static JAL
- RAS

重新打开后，CoreMark 又出现了新问题：

- 不再是之前那种 CRC 错；
- 而是程序再次超时；
- 热点落在一个明显不该无限循环的区域。

这说明：

> BTB-only 修掉的是条件分支恢复问题，但 JAL/JALR 正确预测路径上仍有控制流 bug。

---

## 八、第六步：发现正确预测的 JAL/JALR 仍被 WBU 晚级重定向

再次看 trace 和热点后发现一个结构性问题：

- IFU 已经根据 JAL 或 RAS 提前把正确目标取到了；
- 但到 WBU 时，这些控制流仍然会触发 `pc_update`；
- 等于前端已经走在正确路径上，晚级又重新冲了一次。

这会带来两个后果：

1. 正确预测带来的收益被抵消；
2. 在某些时序组合下，甚至可能破坏控制流节奏，导致死循环或前后级状态错位。

### 根因本质

WBU 把：

- `JAL`
- `JALR`
- `BRANCH`

都当作“无论如何都要晚级更新 PC”的事件处理。

这对于无预测或保守实现可能没问题，但对于已经在 IFU 做了正确预测的系统，会形成 **重复重定向**。

### 修复

WBU 的 `pc_update` 条件改成：

- `ecall/mret` 仍然必须重定向；
- `JAL/JALR/BRANCH` 只有在 `!predict_correct` 时才需要重定向。

这一步修完后，正确预测路径终于不会被晚级平白冲掉。

---

## 九、第七步：x0 旁路污染是另一个隐藏 bug

JAL/RAS 路径修完后，又出现了一次非常早的架构分歧。

现象是：

- 某条 `li a0, 1`
- 理论上应把 `a0` 写成 `1`
- 实际却写成了一个看起来像返回地址附近的值。

这个现象非常像“旁路拿错值”。

继续检查寄存器堆旁路逻辑，发现：

- 旁路判断只看 `rd` 是否匹配；
- 但 **没有排除 `rd == x0`**。

### 为什么这会出错

例如 `ret` 本质上是：

- `jalr x0, ra, 0`

架构上它不会写 `x0`，但执行路径里仍然可能产生一个“结果值”。

如果旁路网络只看：

- `rd == 0`

而不判断“`x0` 不能被真正写回”，那么后续刚好读取 `x0` 或相关寄存器的指令就可能看到一个根本不该出现的值。

### 修复

在 EXU/WBU 旁路条件里增加：

- `rd != 0`

这样 `x0` 不仅在寄存器阵列里恒为 0，在旁路语义上也真正恒为 0。

这一修复对于控制流指令尤其重要，因为很多返回类指令天然就会使用 `rd = x0`。

---

## 十、辅助修补：EXU→WBU flush 窗口处理

除了上述主线 root cause 外，调试中还发现 EXU→WBU 在 flush 周期附近存在旧 valid / 重复提交风险。

因此对 `exu_wbu_regs` 做了更稳妥的 flush/valid 处理，避免：

- 旧提交残留；
- flush 拍错误锁存；
- 错误路径和有效位之间的时序错位。

这项修改不是全部 bug 的唯一根因，但它提升了提交路径稳定性，也让后续 trace 更可靠。

---

## 十一、为什么这次调试方法有效

这次能够比较快收敛，不是因为一开始就猜中了 RTL 问题，而是因为方法对了。

核心方法可以概括为：

### 1. 先建立正确基线

先确认“无预测时 CPU 正确”，避免在一个本来就坏掉的系统上调预测器。

### 2. 逐项启用功能

不要在“全预测全开”的混乱态直接找 bug，而是先拆成：

- BTB-only
- JAL-only
- RAS-on/off

通过隔离实验把问题缩小。

### 3. 以“第一处分歧”为中心

不要只盯最终 CRC 错误，而是追第一条错误提交。

对流水线 CPU 来说，第一处分歧通常最有价值；后面大量错误都只是连锁反应。

### 4. 不把时间相关差异误当根因

例如：

- `mcycle`
- `start_time`
- `stop_time`

这些很容易误导调试方向，必须先过滤。

### 5. 每次只修一个明确 root cause

不是一次性改很多控制逻辑，而是：

- 看现象；
- 定位具体错误提交；
- 找到对应 RTL 条件；
- 做最小修复；
- 再验证。

---

## 十二、最终得到的几个根因类别

这次分支预测 bug 最终并不是单一问题，而是几类控制流相关问题叠加：

### 1. 误预测恢复条件不完整

- predicted taken / actual not taken 没被正确恢复。

### 2. 正确预测路径被晚级重复干扰

- 正确预测的 JAL/JALR/Branch 仍被 WBU 强行重定向。

### 3. 控制流指令对旁路和提交边界产生污染

- `x0` 旁路没有被正确屏蔽。

### 4. flush / valid / commit 边界需要更稳健

- EXU→WBU 在 flush 周期附近需要更严格控制。

这些问题共同说明：

> 分支预测系统真正难的部分，不只是“预测对不对”，而是“预测错了之后，整个流水线如何准确、统一、无副作用地恢复”。

---

## 十三、这次调试对后续架构演进的启发

这次问题虽然表面上属于“分支预测 bug”，但实际上暴露的是更深层的架构问题：

### 1. 提交边界必须足够清楚

执行结果、旁路结果、最终提交结果，三者不能混淆。

### 2. 正确预测路径应尽量少受晚级干扰

否则预测器即使预测对了，也得不到真正收益。

### 3. flush / redirect / kill 语义必须统一

未来如果接入协同向量处理器，这一点会比现在更重要。

### 4. trace 基础设施非常值得保留

`commit-trace` / `mem-trace` 并不是一次性调试工具，而是后续一切复杂微架构演进时最重要的验证手段之一。

---

## 十四、结论

这次分支预测 bug 的发现和修复过程，可以总结成一句话：

> 不是靠猜测 RTL，而是靠“先建立正确对照基线，再做提交级锁步对比，最后围绕第一处分歧回到 RTL 找根因”。

最终修复的关键点包括：

1. 修正 branch mispredict 恢复条件；
2. 给 branch 提交路径传真实 redirect PC；
3. 正确预测的 JAL/JALR/Branch 不再触发多余 WBU redirect；
4. 屏蔽 `x0` 旁路污染；
5. 加固 EXU→WBU flush/valid 语义。

这套过程之所以重要，不只是因为它修好了当前 CoreMark，而是因为它为后续更复杂的 CPU 演进建立了一个非常可靠的调试范式。
